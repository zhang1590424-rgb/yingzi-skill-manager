## 18条核心决策原则

以下决策原则从20期真实评审会中提炼，覆盖了评审中最常用的决策逻辑。在评审方案时可快速对照：

| # | 原则名 | 核心表述 | 应用场景 |
|---|--------|---------|---------|
| 1 | **主场景优先** | 激励承接必须到主场景（feed），不进任务页。任务页预算要下降。 | 所有涉及用户承接路径的需求 |
| 2 | **收益先行** | 先看收益天花板再看成本。做事优先确认收益，再评估投入是否值得。 | 需求优先级排序、方案评估 |
| 3 | **收益门槛** | 砍掉个位数收益需求，聚焦十位数收益。收益不够大就不做。 | 需求筛选与优先级排序 |
| 4 | **增量收益** | 商业化出资需要是增量收益，不能左右腾挪。 | 涉及商业化联合出资的需求 |
| 5 | **成本不兜底** | 活动成本走招商逻辑，实验有时间点后业务方自担。CPLT达标可出，未达标对方出。 | 所有涉及预算分配的需求 |
| 6 | **确定性驱动** | 活动必须有保底奖励。大奖逻辑必须搭配必得逻辑。确定性影响参与动力。 | 所有活动、抽奖类需求 |
| 7 | **轻交互优先** | Feed场景禁弹窗，弹窗负LT。用弹幕、挂件等轻交互。拓展态不能太频繁。 | 所有feed场景内的交互设计 |
| 8 | **用户认知自洽** | 用户看到的状态、进度、规则必须与其行为和预期一致，不能出现认知断裂。 | 状态展示、进度设计、规则设计 |
| 9 | **操作可逆性** | 用户的关闭/隐藏/收起操作必须有明确恢复路径和引导。 | 挂件、入口、功能开关设计 |
| 10 | **数据否决权** | 已在其他端/场景验证为负的方案不应在新端重复尝试。正向结论可复用。 | 跨端方案选择、方案论证 |
| 11 | **控制变量** | 实验中变量变化不能太剧烈，需拆细档位逐步验证。 | 策略实验设计、降本实验 |
| 12 | **供给侧先行** | 需求侧任务需要供给侧先成熟。供给不足不推需求侧。 | 涉及商家/内容/券的任务需求 |
| 13 | **体系简洁** | 不新增专属金币/货币/概念，保持体系简洁。 | 涉及奖励、货币设计的需求 |
| 14 | **上游对齐** | 只要与上游对齐就支持，不能未对齐就自行操作。 | 所有跨团队/跨业务合作需求 |
| 15 | **利益对齐** | 不能用作者成本为平台带收益。利益归属要对齐。 | 涉及作者/商家与平台利益分配的需求 |
| 16 | **玩法核心聚焦** | 一个玩法的任务应围绕核心价值，不应沦为各业务导流容器。 | 玩法/活动任务设计 |
| 17 | **素材承接一致** | 投什么素材就用什么承接。素材对应承接，用户预期一致。 | 投放素材与端内承接 |
| 18 | **自有生态优先** | 能用自家产品/平台解决的，不优先依赖竞品生态。 | 平台选择、跳转路径设计 |

## 15种常见被挑战的问题模式

这些是真实评审会上方案被打回的高频原因。在评审方案时重点关注：

| # | 问题模式 | 表现 | 评审反应 |
|---|---------|------|---------|
| 1 | **成本来源不清** | "预算是谁出的？"答不上来 | 先搞清楚成本归口再来 |
| 2 | **增量vs存量混淆** | 商业化出资看似有收益，实际是左右腾挪 | 需要论证是否为真正增量 |
| 3 | **链路断裂** | 用户进来后不知道去哪，或返回路径不合理 | 画完整链路图补全断裂点 |
| 4 | **需求文档不完善** | 缺背景/缺收益逻辑/缺核心目的/无PRD | 补齐文档再来评审 |
| 5 | **交互过重** | 弹窗→改toast；蒙层→改挂件 | 用更轻的交互方式 |
| 6 | **任务水化** | 太多/嵌套过多/抽奖次数过高 | 精简到核心任务 |
| 7 | **未与上游对齐** | 缺少商业化/主框架/法务的沟通确认 | 先去沟通确认再来 |
| 8 | **文案不实/抬高预期** | 误导性承诺，实际兑现不了 | 修改文案确保真实 |
| 9 | **承接到任务页** | 激励入口承接到任务页而非feed主场景 | 改为承接到feed主场景 |
| 10 | **需求标题太虚** | 标题如"提收"，看不出做什么 | 标题要具体体现做什么 |
| 11 | **供给侧不成熟** | 商家/内容/券不足就推需求侧任务 | 先确认供给侧成熟度 |
| 12 | **新增货币概念** | 引入专属金币/新代币，体系变复杂 | 用已有体系承载，不新增概念 |
| 13 | **无视觉稿** | 没有样式图/demo就上评审 | 补充样式图再评审 |
| 14 | **复用已负向方案** | 其他端验证为负的方案换个端再试 | 放弃，换方案 |
| 15 | **排序用简单系数干预** | 人为给某类任务乘系数提权 | 用多目标优化框架替代 |

## 评审追问清单（按维度）

> 以下维度编号与 SKILL.md 第六章"评审维度速览"一一对应。

### 维度1：收益路径与真实增量

| 追问 | 为什么要问 |
|------|-----------|
| 实验是否能观测到三端去重DAU和LT的正向显著结果？ | 避免只看单端数据被局部正向误导，三端去重才是真实收益 |
| 收益是增量还是左右腾挪？ | 比如商业化出资的翻卡玩法，看起来有收益但不是激励真正的增量 |
| MROI是否大于0？MROI < 0时是否还有做的必要？ | 资源有限，MROI < 0的需求需要特别强的理由才能推进 |
| 是否有大反转实验验证月活收益？ | 对于大功能上线，需要反转实验才能证明因果关系 |
| 收益预估的数据来源是什么？是推算还是实测？ | 拍脑袋的收益预估在评审会上经不起追问 |
| 收益天花板是多少？由什么决定？ | 做事应先看收益天花板再看成本，天花板太低不值得投入 |
| 收益是个位数还是十位数？ | 个位数收益的需求直接砍掉，聚焦十位数收益 |
| 这个数据是单端还是三端？ | 引用收益数据时要区分单端和三端，避免被局部数据误导 |
| 退场后收益还在吗？ | 关注激励退场后的长期效应，是否可持续 |

### 维度2：成本、ROI与止损

| 追问 | 为什么要问 |
|------|-----------|
| 预算是己方还是商业化出资？ | 成本归口必须清晰，说不清则不通过 |
| CPA/CPLT效率如何？ | 核心效率指标，决定方案的经济可行性 |
| 有无无损降本空间？ | 是否做过降本尝试？参考头条/头体的经验 |
| 成本需控制在什么水位？实验期间上限多少？ | 防止实验跑飞、成本失控 |
| 是否走竞价逻辑？ | 各业务方统一走竞价，按CPM/LT导向排序 |
| 降本策略是否无损？会不会影响用户感知？ | 降本可以，但不能让用户明显感知金币变少——否则回到首要原则 |
| CPLT达标部分谁出？不达标呢？ | 精细化成本分担：达标己方出，不达标对方出 |
| 实验期间谁承担？推全后成本切割怎么做？ | 明确时间节点，到点后业务方自担，避免无限兜底 |
| 活动成本能否走招商逻辑？ | 活动成本让招商方承担，自己不出成本就没问题 |

### 维度3：用户体验与客诉风险

| 追问 | 为什么要问 |
|------|-----------|
| 弹窗/蒙层是否过重？能否用更轻的交互替代？ | 重交互打断核心体验，转化不一定更好，但客诉一定更多 |
| 是否有兜底保护？异常状态怎么处理？ | 挂件退场需保护激励进程中和有余额用户；未登录用户怎么办 |
| 用户能理解规则吗？一看就懂吗？ | 霸王餐隔天助力逻辑就是反例——用户理解不了 |
| 文案是否清晰、真实？会不会抬高预期？ | "iPhone17免费领"这种文案绝对不行 |
| 客诉预估量级多少？有无前车之鉴？ | 挂件收起态停转就导致客诉上涨，要提前预判 |
| 频控设置是否合理？同一用户一天会被触达几次？ | 频控失控=骚扰用户 |
| 用户操作后看到的状态是否自洽？ | 用户明明在用但进度为0——认知断裂会引发大量客诉 |
| 用户的关闭/隐藏/收起操作是否可逆？恢复路径是否清晰？ | 操作不可逆且无恢复路径会导致用户"找不到回去的路" |
| 活动有保底/必得奖励吗？ | 纯大奖概率型活动参与动力不足，必须搭配必得逻辑 |

### 维度4：规则/文案可理解性

| 追问 | 为什么要问 |
|------|-----------|
| 用户一眼能看懂怎么做、拿什么、限制是什么吗？ | 规则复杂=用户放弃 |
| 关键边界（资格、上限、冷却、失败原因）是否前置展示？ | 隐藏限制=用户投诉 |
| 文案是否与用户实际体验一致？ | "分百万现金"但人均几分钱——这是欺负用户 |
| 是否存在违反直觉的规则？ | 霸王餐"隔天助力"就是反例，用户根本理解不了 |

### 维度5：实验设计与可归因性

| 追问 | 为什么要问 |
|------|-----------|
| 是否有对照组/反转实验？ | 没有对照就没有因果归因 |
| 实验流量多大？是否需要先小流量再扩量？ | 一上来就全量很危险，尤其是涉及成本的需求 |
| 不同实验组之间是否控制变量？ | 一次改多个变量就分不清哪个有效 |
| AB测试的核心指标是哪些？ | 关注的指标要提前定义，避免事后挑选有利指标 |
| 实验周期够不够长？ | 太短看不到长尾效应，尤其是LT相关实验 |
| 实验期间成本上限怎么控制？ | 防止实验跑飞 |
| 这个方案在其他端/场景有没有验证过？结论是什么？ | 已在其他端验证为负的方案不应在新端重复尝试 |
| 数据是否可靠？是否受节假日/大促影响？ | 春节等特殊时期数据波动大，不能作为决策依据 |
| 是否需要大反转实验？ | 单方向实验可能低估效果，需有无对照实验验证整体价值 |

### 维度6：主场景一致性与承接合理性

| 追问 | 为什么要问 |
|------|-----------|
| 用户点击后落在哪个页面？ | 确认承接场景是否为最优场景 |
| 为什么不进feed而进任务页？ | 任务页不是核心消费场景，承接到任务页会增加成本 |
| 承接路径的成功率是多少？与大盘常规值差多少？ | 量化承接效果，发现承接断裂 |
| 投放素材与端内承接是否一致？ | 素材要对应承接——投什么素材就用什么承接 |
| 用户从这个入口进来后的第一个核心动作是什么？ | 承接页必须让用户立刻能做核心动作 |
| 落地页选择是否匹配用户的核心使用场景？ | 一分购对三端LT负向，换果园后效果好——落地页选择直接影响LT |

### 维度7：机制设计与任务合理性

| 追问 | 为什么要问 |
|------|-----------|
| 任务是否太多太杂？能否精简？ | 任务越多用户越迷茫，转化率越低 |
| 任务完成门槛是否合理？ | 太高→完成率低、用户放弃；太低→被刷、无价值 |
| 抽奖/奖励次数上限是否合理？ | 上限太高→水化；上限太低→用户觉得没意思 |
| 是否与已有任务冲突或重复？ | 新任务可能蚕食已有任务的流量 |
| 任务是否导向核心场景？ | 激励任务要导向看视频（feed），不是导向非核心场景 |
| 这个玩法的核心价值是什么？新增的任务是否服务于核心价值？ | 一个玩法内的任务应围绕核心价值，不应沦为各业务导流容器 |
| 排序逻辑是否合理？是否只看CTR？ | Push和任务页排序不能只看CTR，要纳入LT、任务完成率等指标 |
| 任务展示人群与前置条件匹配吗？ | 对未消费用户展示写评价任务是无效投放 |

### 维度8：套利、作弊与补贴迁移

| 追问 | 为什么要问 |
|------|-----------|
| 是否存在脚本/小号/互刷/代操作风险？ | 社交类机制天然有套利风险 |
| 用户能否"只拿奖励不贡献价值"？ | 奖励设计必须绑定真实行为 |
| 是否存在自然行为迁移到补贴行为的风险？ | 本来就会做的行为被补贴化=浪费钱 |
| 对其他模块是否有补贴虹吸？ | 新激励可能蚕食已有模块的流量和效果 |
| 风控前置了吗？ | 作弊收入退款、准入拦截等需要与风控联动 |

### 维度9：链路完整性与跨团队对齐

| 追问 | 为什么要问 |
|------|-----------|
| 用户从入口到完成任务的完整交互链路是什么？ | 画出完整链路才能发现断裂点 |
| 落地页/承接页设计是否合理？ | 投放来的用户落到哪？任务页还是feed？ |
| 未登录/异常状态下怎么处理？ | 边界case是客诉高发区 |
| 双端（iOS/Android）是否对齐？有哪些差异？ | 双端不一致会带来维护成本和用户困惑 |
| 是否与主框架/商业化/法务/风控沟通过？ | 核心依赖方必须提前对齐 |
| 竞价结算/成本分担机制是否清楚？ | 钱的事情一定要提前说清楚 |
| 技术实现是否与端上同学对齐？ | 有些方案在技术上不可行或成本极高 |
| 利益归属是否合理？是否用他方成本为己方带收益？ | 不能用作者成本为平台带收益，利益归属要对齐 |
| 先定主链路了吗？还是一上来就铺入口？ | 先明确主链路再考虑入口，不要一上来就铺6个入口 |

### 维度10：供给侧成熟度与货币体系

| 追问 | 为什么要问 |
|------|-----------|
| 当前供给侧（商家/内容/商品/券）是否支撑这个需求？ | 供给不足时用户体验会极差 |
| 如果供给不足，用户完成任务的质量如何保证？ | 60%投稿不符要求就是供给不足的信号 |
| 是否需要先扩大供给再推需求侧？ | 供给侧先行，避免需求空转 |
| 发券/发优惠的门槛是否合理？无门槛券vs有门槛券？ | 有门槛券用户不感兴趣，无门槛券才有吸引力 |
| 是否新增了货币/概念？有没有必要？ | 新增货币概念会增加用户理解成本 |
| 能否用现有金币体系承载？ | 优先复用现有体系，不造新概念 |
| 各端货币/概念是否统一？会不会产生水位差？ | 调整金币规则时各端应统一 |

### 维度11：范围控制与一期策略

| 追问 | 为什么要问 |
|------|-----------|
| 一期范围是否过大？能否砍到最小可验证闭环？ | 范围过大→变量过多→失败了都不知道死在哪 |
| 是否同时铺了太多入口？ | 先定主链路再考虑入口扩展 |
| 哪些功能可以挪到二期？ | 能后做的就后做，先验证核心因果点 |
| 失败后能否快速止损回滚？ | 一期设计必须考虑退出成本 |

### 维度12：视觉规范（补充检查项）

| 追问 | 为什么要问 |
|------|-----------|
| 能否提供预期样式图/demo？ | 没有视觉稿很难判断最终效果 |
| icon/标题/副标题的规范是什么？字数限制？ | 避免上线后文字溢出或截断 |
| 多端展示是否一致？跨端文案映射是否统一？ | 跨端不一致会带来客诉 |
| 入口样式是否与已有活动有区分度？ | 复用严重=用户难区分 |

### 维度13：文档规范（补充检查项）

| 追问 | 为什么要问 |
|------|-----------|
| 文档开头讲清背景、目的和收益了吗？ | 没有背景目的的需求无法评判价值 |
| 需求标题是否清楚体现要做之事？ | 标题太虚会被降优先级 |
| 有PRD吗？ | 口头讲需求无法准确评审，必须有书面PRD |
| 收益预估有明确数据支撑吗？ | 每周向上汇报有据可依 |
