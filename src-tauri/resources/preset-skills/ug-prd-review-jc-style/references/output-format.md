# 评审输出格式模板

本文档定义两种评审模式的输出格式。根据用户在第零步选择的模式，使用对应格式。

---

## 深度评审输出格式

评审完成后，严格按以下结构输出：

### 1. 方案一句话判断

例如：
- 值得做，但当前版本还不能进正式评审
- 方向成立，但收益链路和实验设计明显不够
- 不建议做，当前更像高成本换短期表面数据
- 可以推进小范围实验，但必须先砍变量并补齐止损
- 当前最大问题不是机制，而是量级和增量性说不清

### 2. Gating 结果

Gate 1-5 各给出结论和理由。

### 3. 详细评审

按13个维度逐项输出：结论 / 判断依据 / 关键追问 / 修改建议

### 4. 最关键的 3 个生死问题

按优先级排序，列出当前最影响方案成败的问题。

### 5. 建议动作

明确下一步：
- 建议直接推进
- 建议缩范围后重评
- 建议补材料后再评
- 建议先做小流量实验
- 建议先重写实验方案
- 建议暂缓 / 不做

### 6. 如果要进入正式评审，还缺什么

列出必须补齐的材料（收益链路图、增量性论证、人群拆分与量级测算、ROI/预算口径、风险与风控预案、实验方案、成功/失败/扩量阈值、主场景承接方案、规则与文案样稿、视觉稿等）

### 行动建议格式

```
## 行动建议（按优先级排序）
1. 【P0-阻塞项】[具体要做什么] — [为什么这是阻塞项]
2. 【P1-重要】[具体要做什么] — [为什么重要]
3. 【P2-建议】[具体要做什么] — [做了会更好但不阻塞]
```

### 总评结论四种等级

| 结论 | 条件 |
|------|------|
| 🟢 通过 | 方案可执行，直接推进 |
| 🟡 通过有Todo | 大方向通过，需修改具体细节后推进 |
| 🔴 需调整/重新评审 | 方案方向有问题，需重大修改后重新评审 |
| ⏸ Hold/暂不做 | 时机不成熟或前置条件不满足 |

### 7. 产品哲学启发

> 本章节在所有具体评审结论和行动建议之后输出，与前面的评审维度**完全解耦**。从更高的产品视角给出方向性思考，帮助方案负责人跳出细节重新审视方案的底层逻辑。

**选择产品哲学的判断逻辑**：

| 方案特征 | 推荐引用的哲学 |
|---------|-------------|
| 在"用户感知"层面有盲区 | 哲学1：用户可感知的价值、哲学5：不要让用户失望 |
| 范围过大或切口不聚焦 | 哲学2：从小而美到大而全、哲学3：区分目的和手段 |
| 缺少长期牵引设计 | 哲学6：长期目标牵引+短期正向反馈、哲学7：增加确定性 |
| 在成本上陷入死局 | 哲学10：全局收益最大化、哲学11：联合出价与共赢思维 |
| 缺少用户验证 | 哲学9：做好用户调研、哲学8：数据大法+Case大法 |
| 主链路不清晰 | 哲学4：设计好用户的主链路、哲学13："与我有关" |

**输出规则**：

1. **选取2-3条**与当前方案最相关的产品哲学（来自 `references/philosophy.md`），不需要全部引用
2. **每条哲学的输出结构**：
   - 哲学名称（加粗）
   - 一句话点明该哲学与当前方案的关联
   - 1-2个启发性问题，引导方案负责人自行思考
3. **语气**：从评审官切换为产品思考者。不是"你必须改"，而是"你可以想想"
4. **禁止**：不要重复前面已经给过的具体评审意见和修改建议

**输出示例**：

```
## 产品哲学启发

💡 以下是跳出具体评审维度后，从更高产品视角的一些思考：

**从"用户价值"到"用户可感知的价值"**
当前方案提供了不少功能，但用户打开后第一眼能感知到的核心价值是什么？
→ 想一想：如果只保留一个让用户"哇"的点，会是什么？用户需要几步才能感知到这个价值？

**从小而美到大而全**
一期方案覆盖了多个场景和入口，但用户记住的往往是一个具体的打动点，而非全面的能力矩阵。
→ 想一想：能否先找到一个最能打动目标用户的场景，把它做到极致，再考虑扩展？
```

**注意**：产品哲学启发部分的语气不适用 `references/voice-style.md` 的评审语气规则。这部分应当更温和、更启发式，像产品前辈在评审后单独跟你聊的那段话。

## 浅度评审输出格式

浅度评审的输出必须**高度浓缩**，整体篇幅控制在一屏以内。目标是让方案负责人在30秒内抓住关键信息。

### Highlight 结论

用**一句话**给出方案的总判断，直接说清：做还是不做、最大的问题是什么。

这句话必须包含：
- 方案的总体判定（能推进 / 需调整 / 不建议做 / 暂缓）
- 最核心的原因（一个，不要堆砌）

**示例**：
- "收益链路说不清、增量性没论证，先把这个补了再来——不然就是在赌。"
- "方向成立，但一期太重了，砍到最小闭环先验证核心因果点。"
- "本质是存量搬运，左右腾挪换了个口径，不建议做。"
- "没问题，方案清晰、收益可论证、成本可控，按这个推。"

**格式要求**：
- 使用 `> ` 引用格式突出显示
- 句尾标注总评等级：🟢 / 🟡 / 🔴 / ⏸

```
> [一句话结论] — [总评等级]
```

### 三个最关键的问题和建议

从 Gating 结果和对方案的整体判断中，提炼出**最影响方案生死的三个问题**，每个问题附带一条可执行的建议。

**每条格式**：

```
**[问题编号]. [问题一句话概括]**
[为什么这是关键问题——1-2句话说清风险/影响]
→ 建议：[具体要做什么]
```

**约束**：
- 严格只输出三条，不多不少
- 按优先级排序（最致命的排第一）
- 问题描述要具体，不要泛泛说"收益需要再看看"，要说清"收益的什么方面有什么问题"
- 建议必须可执行，不要说"建议优化"，要说"做什么、怎么做"
- 每条问题+建议控制在3行以内
- 语言风格同样遵循 `references/voice-style.md`：直接、有方案、不绕弯

**输出示例**：

```
**1. 收益是增量还是存量搬运？**
当前收益口径把用户从自然路径迁到激励路径的部分也算进去了，这就是左右腾挪。
→ 建议：补一组对照实验，把自然转化基线扣掉，看纯增量还剩多少。

**2. 成本口径不清，ROI算不过来**
预算谁出、最坏情况烧多少、止损线在哪——三个都没写。
→ 建议：先跟商业化对齐成本归口，定好止损线和熔断阈值，再来评。

**3. 一期范围太大，失败了都不知道死在哪**
同时铺了4个入口+3种任务+2个玩法，变量太多无法归因。
→ 建议：砍到1个入口+1个核心任务，先验证"用户会不会因为这个激励多看一条视频"这一个因果点。
```

### 浅度评审补充说明

1. **不输出 Gating 逐项结果** — 浅度评审的 Gating 分析是内部推导过程，不直接呈现，只体现在最终的三个问题中
2. **不输出13维度评审** — 这是深度评审的内容
3. **不输出产品哲学启发** — 浅度评审聚焦实操问题，不展开哲学层讨论
4. **不输出缺失材料清单** — 如果材料缺失是关键问题，放进三条建议里
5. **语言风格一致** — 浅度评审同样遵循 `references/voice-style.md` 的语气规则，甚至应该更直接、更精炼
