## 模式一：利益方诉求映射

### 什么时候用
当文档涉及多方协作、多团队对齐、平台治理等场景时，在提出任何框架或方案之前，先把各方的诉求和期望摆出来。

### 怎么做
为每个关键利益方建立三要素卡片：

```
[利益方名称]
- 核心诉求：他们想要什么（1-3条）
- 期望模式：他们希望怎么合作
- 能力/资源：他们能带来什么
```

### 为什么有效
- **建立共情**：读者看到"你理解我的处境"，信任感立刻上来
- **暴露矛盾**：各方诉求放在一起看，冲突点自然浮现，为后面的框架铺路
- **防止偏颇**：避免文档只站在一方立场（小环境单一化），增强"新加坡定位"的可信度

### 实战案例
在「关于开放还是闭环的业务思考」中，作者在提出开放/闭环框架之前，先逐一列出了生活服务、小程序、懂车帝三方的合作诉求和期望模式。读者看完立刻理解：为什么需要一个统一框架——因为各方诉求差异巨大，需要一个共识基础来协作。

## 模式二：概念分类表

### 什么时候用
当文档需要引入新概念、新分类、新框架时——任何需要"建立共同语言"的场景。

### 怎么做
用结构化表格定义每个概念的三要素：

| 分类 | 定义（一句话说清楚） | 关键特征（区分于其他分类的要素） |
|---|---|---|
| 类别A | ... | ... |
| 类别B | ... | ... |

### 操作要点
1. **穷尽且互斥**：分类应覆盖所有情况，且彼此不重叠
2. **附带真实案例**：每个分类至少配一个大家认识的案例
3. **标注业务状态**：如果分类之间可以转换（如从开放到闭环），标注转换路径

### 为什么有效
- 把模糊概念变成精确定义，消除沟通歧义
- 表格格式让对比一目了然
- 建立的分类体系成为后续讨论的"棋盘"——所有人在同一个棋盘上下棋

### 实战案例
在「关于开放还是闭环的业务思考」中，作者用一张9行3列的表格，将"开放"和"闭环"从模糊概念拆解为7种具体状态（开放/内部开放/外部开放/闭环/开闭并存/切闭环/半闭环），每种都有明确定义和关键要素。这张表让所有参与讨论的人有了共同的坐标系。

## 模式三：反面案例教学

### 什么时候用
当你需要定义"什么是好的"时，先展示"什么是坏的"——用已知的失败案例来划清边界。

### 怎么做
1. 选择受众已知且认可的失败案例
2. 展示因果链：做了什么 → 为什么出了问题 → 造成了什么后果
3. 从反面推导正面：所以"好的做法"应该避免这些、做到这些

### 模板
```
[反面案例标题]
过度/不当的 [做法] 导致了 [具体问题]：
- [案例1]：[做了什么] → [后果是什么]
- [案例2]：[做了什么] → [后果是什么]

因此，好的 [做法] 应该：[正面定义]
```

### 为什么有效
- 人对"避免损失"的动力远大于"获得收益"（损失厌恶）
- 具体的失败案例比抽象的原则更有说服力
- 让读者自己得出结论：先看到问题，再看到解法，比直接给解法更有认同感

### 实战案例
在「关于开放还是闭环的业务思考」中，定义"好的开放"时先列出反面案例：早期电商跳淘宝（没有购买数据、评价、物流反馈不透传）和当前小程序支付（核心交易无法全部管控）。读者看完自然理解：好的开放=开放能力的同时建设好数据互通和管控机制。

## 模式四：真实演进佐证

### 什么时候用
当你提出的框架或分类看起来"太理论化"时，用已有业务的实际演进路径来证明框架的实用性。

### 怎么做
选取2-3个受众熟悉的业务，用一句话描述它们在你框架中的位置和演进轨迹：

```
- 业务A：选择了 [路径X]（例如...）
- 业务B：从 [路径X] 走向了 [路径Y]，但保留了部分 [路径X]
- 业务C：当前处于 [混合状态]
→ 说明不同业务在不同阶段会做出不同选择，框架是通用的
```

### 为什么有效
- 理论有了实践锚点，可信度飙升
- 受众可以对号入座：我的业务像哪个案例？我可能走哪条路？
- 展示了灵活性：框架不是非此即彼的死规矩，而是可以根据实际情况选择和演进

### 实战案例
在「关于开放还是闭环的业务思考」中：电影票（暂时开放，支持猫眼）、电商（从开放到闭环，保留部分开放）、生活服务（部分开放+半闭环）。三个案例把抽象的"开闭选择"变成了可触摸的业务现实。

## 模式五：分层架构递进

### 什么时候用
当文档需要从宏观框架落到具体执行时——典型场景：战略→原则→建设方案。

### 怎么做
三层递进结构：

1. **框架层**：定义是什么（概念、分类、全景图）
2. **原则层**：定义好的标准是什么（成功准则、红线、关键判断依据）
3. **执行层**：具体要建设什么、谁来做、怎么做

### 操作要点
- 每一层独立成段或成章，不要混在一起
- 上层为下层提供判断依据：为什么这个执行方案是对的？因为它符合原则层定义的标准
- 下层为上层提供落地路径：原则层定义的标准，通过这些具体措施来实现

### 为什么有效
- 满足不同层级受众的需求：决策层看框架和原则，执行层看具体建设方案
- 逻辑链清晰：框架→原则→执行，层层递进，没有跳跃
- 便于讨论聚焦：可以分层讨论，不会把"要不要做"和"怎么做"混在一起

## 模式六：排版视觉层级

### 核心原则
排版不是装饰，是信息架构的视觉表达。好的排版让读者在 3 秒内看清文档结构、10 秒内找到自己关心的部分。

### 飞书文档排版规范

| 元素 | 用途 | 示例 |
|---|---|---|
| **引用块（blockquote）** | 补充信息、背景材料、非主线但有价值的内容 | 各方诉求明细、案例补充 |
| **表格** | 结构化对比、概念分类、数据展示 | 模式定义、风险矩阵 |
| **加粗** | 段落内强调关键词、关键结论 | 核心数字、核心结论 |
| **分割线（---）** | 主章节之间的视觉分隔 | 大章节之间 |
| **任务列表（- [ ]）** | 行动项、待办事项 | 会议结论的 next steps |
| **导航索引** | 长文档（>2000字）顶部放章节导航 | > 文档导航｜章节1 · 章节2 · ... |

### 操作准则
1. **主线用正文，支线用引用块**：引用块（blockquote）是"降一级展示"的信号，读者可以选择跳过
2. **一张表只讲一个对比**：避免超宽超长的万能表，拆成多张小表
3. **关键数字必须加粗**：定量数据是弹药库的核心，必须在视觉上突出
4. **行动项用 checklist**：待办事项不是叙述性内容，用 `- [ ]` 格式增加可操作性
5. **长文档必有导航**：超过 2000 字的文档，顶部加一行引用块做章节索引
6. **章节间留白**：主章节之间用分割线 + 空行，给读者喘息空间
7. **标题即观点的视觉强化**：观点式标题比目录式标题更需要视觉层级来帮助快速扫描
