## 核心模型

**故事线 = f(信息量 × 位置 × 目的)**

同一件事情，研发leader、战略同学、产品负责人会写出完全不同的故事线。这不是谁对谁错，而是因为：
- 每个人掌握的**信息量（上下文）**不同
- 每个人**在整个格局中的位置**不同
- 每个人**希望通过这个叙事达成的目的**不同

所以写故事线之前，最核心的问题是：**你的位置是什么？你对大环境和小环境的判断是什么？** 位置决定目标，目标决定故事线。

## 大环境详解

大环境 = 火星视角 = 脱离自身位置看全局。

核心问题：**受众关心的事情中，哪些已达成共同认知、哪些没有？**

- **已达成共同认知的事**：一句话带过。别花篇幅论证大家都同意的事——那是浪费注意力。
- **未达成共同认知的事**：这才是主战场。你需要决定以什么方式给受众建立认知——这个"以什么方式"本身就是你的目的。
- **大家不觉得是问题的问题**：最难也最有价值。如果你能把一个大家没意识到的问题变成共识，你就掌握了叙事的主动权。

### 受众分析工具：三类受众模型

分析大环境时，关键要识别受众类型。不同类型的受众对信息的需求完全不同：

| 受众类型 | 占比参考 | 关注什么 | 信息密度要求 |
|---|---|---|---|
| **管理者**（决策层） | ~5% | 全局判断、资源分配、战略方向 | 高度压缩，只要结论和关键线索 |
| **协作者**（执行层） | ~75% | 跟自己的关系、行动指引、协作边界 | 中等，需要理解逻辑但不需全部细节 |
| **信息同步者**（旁观层） | ~20% | 发生了什么、跟大局的关系 | 较低，知道大方向即可 |

**关键决策**：你的文档首先服务哪类受众？这决定了信息组织的方式。给管理者看的材料如果写成了给协作者看的详细执行方案，管理者会失去耐心；给协作者看的如果太抽象，他们不知道该做什么。

### 需求视角 vs 供给视角

写文档有两种视角——

**供给视角**（以作者为中心）：我做了什么 → 我遇到了什么问题 → 我要什么支持。这是大多数人的默认写法，但它的问题是读者要自己翻译"这跟我有什么关系"。

**需求视角**（以受众为中心）：受众关心什么 → 在这个关心点上发生了什么 → 对受众意味着什么。这种写法让读者觉得"这就是写给我看的"。

诊断时，检查文档是从供给视角还是需求视角出发的。大多数需要优化的文档都是供给视角——作者在讲自己的故事，没有站在受众的位置上思考。

### 2.5维度叙事法

当文档需要在"全局概览"和"深度分析"之间取得平衡时，可以用2.5维度结构：

- **维度1（主线）**：业务全景，用最精简的方式讲清楚整体格局和核心逻辑
- **维度2（拆解）**：按增长公式或关键模块拆解，每个模块讲清楚进展、问题、下一步
- **维度2.5（深钻）**：选取一个最有代表性的案例或模块，深入展开，以小见大

为什么不是维度3？因为给决策层的材料，深钻到-1层就够了。再深入就是执行细节，应该放在附录或单独文档里。

**诊断方法**：审视文档开头3段——
1. 作者花了多少篇幅在论证已有共识？（超过30%大概率有问题）
2. 是否识别出了"受众不知道自己不知道"的盲区？
3. 文档是在回答受众心里的问题，还是在回答作者自己的问题？
4. 叙事深度是否匹配受众类型？

## 小环境详解

小环境 = 地球视角 = 把自己放进格局里，看清自己是谁、周边是谁。

### 新加坡定位法

在复杂的多方利益格局中，最有影响力的位置不是某一方，而是能理解所有方、被所有方认可的中间位置。这不是骑墙——因为你两边都懂，所以你讲的话两边都愿意听。

好的小环境定位：从大家共同认可的案例出发，讲一个大家能接受、觉得有道理、能解决当前问题的新秩序。新秩序被充分认同后，各方会自己对号入座。

**诊断方法**：
1. 作者立场是否只代表一方？如果是，其他角色会觉得"这跟我有什么关系"
2. 是否展现了对多方诉求的理解？如果没有，说服力打折
3. 是否找到了各方都能接受的叙事出发点？

### 利益方诉求映射（强化技巧）

在复杂的多方协作场景中，不要直接跳到方案。先把各方的诉求和期望显式列出来，让所有人看到全貌。这个动作本身就在建立"新加坡定位"——你对所有方都有理解。

操作方法详见 `references/best-practices.md` 的"模式一：利益方诉求映射"。

### 影响决策逻辑而非替人决策

有时我们不是决策者，我们要做的是理清决策者的决策逻辑，然后去影响它。不是替老板决定做与不做，而是把客观事实和逻辑线索摆清楚，引导他做决策。如果他听懂了，他会自己做出合适的决策。

**判断标准**：一个好的故事线 = 给100个人看，100个人能得出跟你一样的结论。目的不会写出来，目的是藏在心里的。

## 目标详解

### 灵魂拷问x3

动笔之前，强烈建议拷问自己三次：

1. **为什么要做这件事？**——搞清楚写这篇文档的真正原因，不是表面需求
2. **谁想看？**——明确关键受众，不同角色的预期不同
3. **为什么要看？**——受众看完想得到什么，想做什么决策

这三个问题看起来简单，但大量文档的问题恰恰出在"没搞清楚预期就开始写"。接收到的需求和实际诉求之间常常有偏差——就像用户说要"一匹更快的马"，真正要的是"更快到达目的地"。

### 好的目标 vs 坏的目标

- ✗ "让大家知道我们做了什么"——这是汇报，不是叙事
- ✗ "让领导批准方案"——太直白，容易引起抵触
- ✓ "构建一个认知框架，让所有人在这个框架下自然得出你希望的结论"

### 预期校准

搞清楚预期，才能达到和超出预期。没搞清楚预期就执行，一开始就是错的。如何搞清预期？
- **多想**：多想两次为什么和怎么做
- **多问**：信息量不够就去问，不要脑补
- **验证**：有了初步想法，形成框架性文档，先确认方向对不对

## 故事线框架怎么写

### 基本要求
- **300-400字足矣**。故事线不是目录大纲，是叙事骨架。
- **用演绎逻辑**：因为A → 所以B → 因此C。不是"第一点...第二点...第三点..."
- **关注"为什么要做"**，不讲"怎么做"——执行细节不是故事线层面的东西。

### 标题写观点，不写目录

这个原则极其重要：

**检验方法**：假设只有一分钟，阅读文档的所有标题——是否能了解全部核心观点？

- ✗ 目录式："一、背景""二、方案""三、数据"——读完完全不知道要说什么
- ✓ 观点式："获客成本持续上升，传统渠道已到天花板""AI投放将ROI提升至2.5x""三个月内可验证效果并回收投入"

观点展示在段头，便于信息获取：如果读者认同结论，阅读结束；如果存疑，继续下钻看论据。这就是金字塔原则——结论先行，上层概括下层。

### 逻辑拆解框架

写故事线框架时，核心是在做逻辑拆解。常用的拆解方式：

**通用逻辑**：现状 → 问题 → To Do（适合把一件事讲清楚）

**业务逻辑**：按增长公式拆解核心指标，围绕指标讲进展、问题、下一步（适合业务汇报）

**分层架构**：框架层（是什么）→ 原则层（好的标准是什么）→ 执行层（具体做什么）。适合战略性文档或平台建设方案。详见 `references/best-practices.md` 的"模式五：分层架构递进"。

关键是：拆解出指标后，所有讲的事情都应该和核心指标相关。如果某段内容跟核心指标没关系，要么它不应该出现在这里，要么你的指标拆解有遗漏。

### 篇幅分配参考
- 建立共识 / 抓注意力：10-20%
- 构建新认知 / 展示格局：40-50%
- 引导决策 / 新秩序：30-40%

## 弹药库

### 核心区分：主线 vs 弹药库

- **故事主线**：讲"国际局势"——全局格局、秩序、各方关系
- **弹药库**：具体的业务逻辑拆解——数据、案例、流程细节

弹药库是为主线服务的论据，不是主线本身。常见错误是把大量篇幅花在业务逻辑细节上，读者看到一棵棵树而看不见森林。

**判断方法**：删掉某段具体业务分析，故事还能不能讲通？如果能，它是弹药而非主线。

### 先有观点，再组织论据

这个顺序至关重要：**要先有业务逻辑和观点，再组织数据和用研信息，整理出支撑观点的论据。** 而不是先有数据和用研，从中找信息拿来用。

如果数据推翻了你的观点，那就围绕新观点重新组织——这恰恰说明观点先行的流程是有效的，数据起到了校验的作用。

### 反面案例作为弹药（强化技巧）

定义"什么是好的"时，先展示"什么是坏的"——用已知的失败案例来划清边界。这种"反面教学"比正面论证更有说服力，因为人对损失的敏感度远高于收益。

操作方法详见 `references/best-practices.md` 的"模式三：反面案例教学"。

### 真实演进佐证（强化技巧）

理论框架容易显得空洞。用真实业务的演进路径来佐证——"A选择了路径X，B从X走到了Y，C目前处于混合状态"——让受众看到框架是从实践中来的，不是拍脑袋。

操作方法详见 `references/best-practices.md` 的"模式四：真实演进佐证"。

### 数据偏差意识

数据是有偏差的，不同统计方法、缺少因素考量都会产生偏差。使用数据作为弹药时保持警觉：
- 幸存者偏差：看到的样本不代表全貌
- 框架效应：同一数据的不同描述会引发不同判断（"90%存活率" vs "10%死亡率"）
- 用研偏差：用户说的、用户以为自己说的、你听到的可能都不一样

### 定量替代定性

- ✗ "方案A显著好"
- ✓ "方案A在xx数据上提升30%"
- ✗ "几乎全部用户"
- ✓ "87%的付费用户"

## 表达技巧

### 快速吸引注意力
**把大家习以为常、又觉得模糊的事情用一句话讲清楚。** 每个人心里都有一个模糊认知，你一旦讲清楚了，他们会觉得"对就是这个意思"，注意力立刻来了。

### 概念分类建立共同语言
当文档需要引入新概念或框架时，用结构化表格明确定义每个分类的定义和区分特征。这个动作把模糊讨论变成精确对话——所有人在同一个坐标系上沟通。详见 `references/best-practices.md` 的"模式二：概念分类表"。

### 提炼共鸣引发联想
从受众关心的事情中提炼有洞察的观点。你提供一个高质量输入，受众会产生大量联想——联想越多越觉得你厉害，越愿意继续听。这需要信息量全面广泛 + 深度思考，没有捷径。

### 图表只讲一个故事
- **表格**突出对比（A vs B、现状 vs 目标）
- **图**讲清楚多维关系（角色关系、流程闭环）
- 人的认知是一维线性的，一张图/表只讲一个故事

### 用成熟案例讲新秩序
想建立新认知框架时，不要空讲理论——找一个大家已认可的成熟案例来类比。读者会自然完成从已知到未知的认知迁移。

### 信息压缩
金字塔原则：结论先行，上层概括下层，同层独立穷尽。麦肯锡电梯30秒理论：最简短高效的沟通方式。如果中国政府的工作报告能用500字说清楚，你的文档大概率也可以。

## 写作模式工具箱

以下是从最佳实践中提炼的六种可复用写作模式，使用场景和操作方法详见 `references/best-practices.md`：

| 模式 | 适用场景 | 一句话说明 |
|---|---|---|
| **利益方诉求映射** | 多方协作、平台治理 | 先摆各方诉求，再提框架 |
| **概念分类表** | 引入新概念、建立共同语言 | 用表格把模糊概念变精确定义 |
| **反面案例教学** | 定义"好"的标准 | 先展示"坏"的样子，反推"好"的定义 |
| **真实演进佐证** | 验证框架实用性 | 用已知业务路径证明框架可行 |
| **分层架构递进** | 战略→原则→执行 | 三层逐级展开，满足不同层级受众 |
| **排版视觉层级** | 所有输出文档 | 排版即信息架构，3秒看清结构 |

## 排版与视觉层级

排版不是最后的装饰步骤，它是信息架构的视觉表达。好的排版让读者在 3 秒内看清文档结构、10 秒内找到自己关心的部分。

### 飞书文档排版规范

| 元素 | 用途 | 何时使用 |
|---|---|---|
| **引用块** | 补充性信息，非主线内容 | 各方诉求明细、案例补充、背景说明 |
| **表格** | 结构化对比和分类 | 概念定义、风险矩阵、数据对比 |
| **加粗** | 强调关键词和关键数字 | 核心结论、关键指标 |
| **分割线** | 主章节间视觉分隔 | 大章节转换时 |
| **任务列表** | 行动项和待办 | 会议结论、next steps |
| **导航索引** | 长文档快速定位 | 超过2000字的文档 |

### 排版操作准则
1. **主线正文，支线引用块**：引用块是"可选阅读"的信号
2. **一表一对比**：避免大而全的表，拆成多张单一焦点的小表
3. **关键数字加粗**：定量数据是弹药库核心，必须视觉突出
4. **行动项用 checklist**：用 `- [ ]` 格式代替普通列表
5. **长文档必有导航**：2000字以上在顶部加引用块做章节索引
6. **章节间留白**：分割线 + 空行给读者喘息空间
7. **观点标题视觉层级**：观点式标题配合加粗和层级缩进，帮助快速扫描

## 常见问题诊断清单

| 症状 | 可能原因 | 优化方向 |
|---|---|---|
| 读完不知道"so what" | 缺少目标或目标不聚焦 | 明确希望读者得出什么结论，反向设计叙事路径 |
| 很长但没有记忆点 | 罗列信息而非演绎逻辑 | 找到核心论证链，删掉不在链上的内容 |
| 开头就失去兴趣 | 在论证已有共识 | 用"讲清楚一个模糊概念"来开场 |
| 不同角色反应差异大 | 小环境定位偏向一方 | 找到"新加坡位置"，用各方都能接受的视角 |
| 业务分析详细但说服力弱 | 业务逻辑占据主线 | 把业务逻辑移到弹药库，主线讲格局和秩序 |
| 结论太突兀不买账 | 目的暴露太直白 | 铺设认知路径，让读者自己走到结论 |
| 图表多但读者跳过 | 图表承载太多信息 | 每张图/表只讲一个故事 |
| 框架像目录 | 标题在列主题不是观点 | 改为观点式标题，读标题就知道结论 |
| 数据很多但不可信 | 缺乏判断力，被数据牵着走 | 先有观点再组织论据，注意数据偏差 |
| 预期总是对不齐 | 没搞清楚受众真正想要什么 | 灵魂拷问x3 + 用框架先行确认方向 |
| 排版混乱找不到重点 | 没有视觉层级 | 主线正文+引用块降级+表格分类+数字加粗 |
| 多方协作文档推不动 | 各方觉得跟自己无关 | 先做利益方诉求映射，再提方案 |
| 新概念讲不清 | 缺乏共同语言 | 用概念分类表建立共同坐标系 |
