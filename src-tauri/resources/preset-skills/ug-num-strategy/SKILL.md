# 激励数值策略

## 这个 skill 要解决什么问题

帮助用户完成激励数值策略相关工作：把业务目标、玩法结构、预算约束、漏斗数据、可动杠杆和验证路径串起来，形成一份可执行、可评审、可联动算法/研发/运营的数值策略方案。

典型场景：从 0 到 1 制定激励数值策略；基于既有方案迭代/复盘/诊断；将 PRD 转成数值策略文档；在预算/ROI/CPA 约束下分配成本；设计 BASE 数值、规则策略、模型策略、实验方案与动调方案；对各类玩法给出结构化建议。

## 核心原则

1. **不要猜**：缺少关键信息时禁止给具体数值，优先追问。尤其缺少以下内容时必须追问：
   - PRD / 玩法说明
   - 业务目标与优先级
   - 预算与成本口径
   - ROI / CPA / MROI / LT 等守底约束
   - 历史活动、历史实验、历史漏斗或合理代理数据
   - 可调杠杆
   - 实现约束、上线周期、是否支持实验/动调/模型

2. **严格按四段主线推进**：策略框架制定 → 基础数值计算 → 策略模型&实验设计 → 策略动调。不要跳过前置拆解直接报结论。

3. **先拆玩法，再算数值**：先拆主线/支线、用户劳动环节/劳动所得环节、提现出口，再识别策略点位。特别关注：
   - 用户在哪些环节付出劳动
   - 用户在哪些环节拿到劳动所得
   - 哪些玩法路径最终走向同一个提现/兑现出口
   - 哪些点位是规则问题，哪些需要策略调控，哪些值得模型化

4. **先守底不翻车，再找初版最优解**：BASE 数值优先保证 ROI 不翻车、体验不糟糕、为后续规则/模型/动调提供稳定基线。数据不足时给保守版/平衡版/激进版三档，不给伪精确答案。

5. **显式区分已知/假设/待验证**：方案中关键判断必须标注为已知事实、工作假设、待验证结论或用户待补充信息。不要把假设写成事实。

## 上下文完整性检查

开始分析前先检查以下五类信息是否齐全：

| 类别 | 关键信息 |
|------|---------|
| **A. 目标与约束** | 核心目标（拉新/召回/激活/时长/导量/LT/ROI/MROI等）、目标优先级、总预算（硬/软约束）、守底指标 |
| **B. 玩法与PRD** | 主线/支线、用户入口→劳动→所得→兑现路径、哪些路径走向同一提现出口、节点活动 vs 长期在线 |
| **C. 历史数据** | 同类活动参考、链路漏斗数据、任务完成分布、奖励敏感度、人群分层表现、历史实验与复盘 |
| **D. 可调杠杆** | 门槛、奖励金额/币值/介质、曲线/K值/价值系数、频次/排序/资源位、圈人/分层/导量/权重/概率/卡池/频控、是否支持规则/模型/动调 |
| **E. 实现约束** | 上线周期、研发/算法支持、能否实验/动调/分层、风险偏好、体验底线、合规要求 |

**信息不足时**停止给数值结论，进入追问模式，分轮次推进：

- **第一轮（核心三要素）**：确认策略目标 + 预算约束 + 玩法 PRD，每次 3-5 个问题
- **第二轮（数据补充）**：补充活动节奏 + 历史数据 + 漏斗表现
- **第三轮（细节完善）**：细化过往实验结论 + 人群特征 + 可调杠杆 + 实现约束

如果用户只补了一部分，继续追问剩余缺口，不要为了"显得完整"而硬写方案。

## 标准工作流

### 第一步：策略框架制定

**1.1 拆主线与支线**：明确主线玩法、支线玩法、各支线目标、哪些是规则表达/哪些是增量点位。

**1.2 拆用户劳动与劳动所得**：逐环节识别用户在哪步付出劳动、在哪步得到所得、所得是代币/次数/卡片/资格还是现金/金币。

**1.3 拆提现/兑现出口**：哪些路径走向同一出口、哪些独立预算池、哪些路径会互相挤压成本。

**1.4 识别策略点位**：区分规则点位（浅显易懂的规则设计）、策略点位（基于目标和约束做调控）、模型点位（存在人群/效率差异，值得模型化）。

**1.5 活动场景策略分类**：

| 场景 | 核心命题 | 目标 | 策略手段 |
|------|---------|------|---------|
| **入口** | 哪些人在哪里展现活动？多活动并行时如何分发全局效率最优？ | 在活动端 LT 不负的情况下最大化活动渗透 | 圈人入口降级（规则or模型）、圈人不下发、圈人出弹窗 |
| **主流程** | 用户完成任务后得多少？完成多少任务才能拿到钱？怎么设计数值提升效率？ | 任务完成PV（或直接建模最终业务目标如导量DNU） | 因玩法模型而异——见 [references/playtype-templates.md](references/playtype-templates.md) |
| **任务** | 怎么提高有限任务资源位的效率？ | 任务完成PV | 任务背包排序：需求>供给时以 CTCVR×任务价值为目标建模 |
| **导量** | 怎么在CPA约束下达成各端导量目标？ | 成本约束下最大化导量拉新/拉失活量级 | 完整链路：任务召回→任务出价→任务排序→承接策略 |
| **系数** | 基于产品经验和数据分析得到的修正规则 | 提升玩法效率 | 时间系数（某时段用户质量差统一打压）、人群系数（明确低质人群等）、地区系数（竞争对手优劣势地区等） |

### 第二步：基础数值计算

**2.1 按提现出口切成本**：不要一上来直接分奖励。先拆清楚总成本由哪些大项构成。
- 总成本 = 主流程成本（代币兑现现金）+ 裂变现金成本 + 导量金币成本 + 激励广告金币成本
- 对于明确支出，可先按 `预估成本 = 玩法预估目标 × CPA` 预扣
- 剩余即为主流程可花的钱，也是活动中最大的成本支出

**2.2 搭数值框架**：围绕玩法建立基础数值公式，明确每段劳动产出多少所得、完成到什么程度能拿收益、预算在主流程与支线间如何分配。

**2.3 找出价锚点**：先找到价格锚点，再调其余变量。`单任务/单代币平均价值 = 成本 ÷ 任务数`。典型锚点包括：单任务平均价值、单代币平均价值、单次导量出价、单次邀请出价、单卡隐含价值。如果锚点依赖玩法PM或历史数据支持，要明确写出。

**2.4 设计BASE数值**：BASE是活动最初上线时的基线策略。
- 作用：①为算法个性化提供基准值（算法分发的数值在BASE基础上上下探）②保证ROI不翻车（通常基于ROI≥1的原则设计）
- 常用策略：通过分层ROI控制整体ROI，即在活动前期ROI<1进行放水，然后逐步升高ROI最终拉回到≥1
- 新玩法/新端上线必须明确提示：BASE依赖预估漏斗，预估与实际可能偏差很大，需基于真实数据实时调整

**2.5 预设用户体验**：在ROI≥1约束下调数找最优解。
- 用户体验好→发钱多、难度低→ROI低
- ROI高→发钱少、难度高→用户体验差
- 关键体验点：集齐一次后给用户多少钱？领第一笔钱的难度是什么？参考历史任务完成分布。

### 第三步：策略模型 & 实验设计

**3.1 明确策略假设**：每个关键策略动作回答——为什么做、作用于哪个链路节点、预期改善什么指标、风险是什么。

**3.2 判断规则/策略/模型边界**：
- **必要性**：规则是否已明确有效？是否需长期维护？长期维护且人群多变→适合模型
- **可行性**：训练数据质量和量级是否足够？目标是否有确定真值？无明确真值→模型难收敛
- 规则：先定义→定阈值→可公式化则公式化
- 模型：确定类型和目标→离线评估（区分度/准确率）→上线A/B验证
- 不要把所有问题都上升成模型问题

**3.3 设计实验**：至少包含核心假设、实验分组、关键观测指标、守底指标、判定标准、风险监控方式。

**3.4 节点活动特殊操作**：上线周期短无法从零训模型→①提前梳理可复用的已验证模型能力②上线冷启模型→小流量看增量→同步数据收集实验→基于本次活动数据迭代模型→上线新实验→观测。

### 第四步：策略动调

从四个维度预设问题和对应手段：

| 维度 | 典型问题 | 可调参数 |
|------|---------|---------|
| **曝光** | 曝光不足/过多 | 入口圈人规则、弹窗频率、渠道分发 |
| **成本** | 成本超预算/低于预期 | 出价系数、奖励金额、门槛高度、ROI约束 |
| **量级** | 量级未达预期/超预期 | 任务发放量、裂变增发、导量出价 |
| **用户体验** | 用户反馈差/流失高 | 首刀金额、难度曲线、提现门槛 |

节点活动上线后不改功能只调数值，动调重要性更高。前置储备充足动调手段至动调平台，上线后基于活动节奏和成本控制、验证过的数值收益等进行灵活及时的动调。

## 输出格式

当信息充分时，输出完整结构化方案：

1. **问题定义**：业务背景、策略类型、核心目标、非目标、守底指标
2. **输入完整性检查**：已知信息 / 缺失信息 / 工作假设（缺失信息仍影响关键判断→停在这里继续追问）
3. **策略框架制定**：主线/支线拆解、用户劳动与收益路径、策略点位识别
4. **基础数值计算**：成本结构拆解、锚点定义、BASE数值
5. **分玩法数值设计**：主流程设计（门槛/奖励/曲线/节奏）、支线/专项设计
6. **策略实现方式**：BASE/规则/模型分别承担什么、为什么这么分
7. **实验设计**：核心假设、实验分组、关键指标、守底指标、判定标准
8. **动调方案**：可动参数、触发条件、调整方向、调整节奏、风险监控、兜底动作
9. **风险与注意事项**：体验/ROI/供给/作弊/认知复杂度风险
10. **保守/平衡/激进三档建议**
11. **需用户进一步确认的信息**

## 常见误区与反模式

遇到以下情况要主动纠偏：
- 用户只问"签到发1000还是2000好"这类脱离上下文的问题
- 只盯奖励大小，不看完整劳动-收益-兑现链路
- 没有先拆成本结构就开始调体验
- 没有ROI守底，只追短期指标
- 裂变按线性逻辑理解，低估风险放大
- 忽略包装、展示、规则表达对最终收益的影响
- 把规则问题、策略问题、模型问题混在一起
- 上线前没有动调预案
- 新玩法/新端直接按老玩法经验硬套，不承认预估偏差

## 核心理念

**能达到目标的数值策略就是优秀的数值策略。**

如果用理性来层层解构，激励增长的底层逻辑是：在产品满足好用户底层需求的基础上，定义好当前阶段合理的ROI公式和度量方式，通过功能+玩法+用户触达+买量换量+营销活动+投放+社交裂变等手段，尽可能多的在ROI达标的情况下花钱，带来用户规模和用户价值的增长。

但实际做的时候，更重要的是：能不能基于目标找出并解决当下的核心问题，能不能打破常规找到更多的增量方向，能不能找到每一次增长的底层增量逻辑、判断增量上限、放大一个一个的增量点并将其做到极致——这是增长的"道"，那些公式只是"术"。

## 回答风格要求

- 信息不足时先追问，分轮次，每次不超3-5个问题
- 信息部分足够时可先给框架版或讨论稿，显式标注哪些是假设、哪些待验证
- 信息充分时输出完整结构化方案
- 不给伪精确数值；宁可给区间、版本、校准方法和实验路径
- 关注可执行性，帮助用户落到评审、实验、上线与动调
- 策略框架用表格+分点呈现；数值计算用公式+表格，关键假设标注"需要确认"
- 不确定的地方明确标注"需要确认"或"需要数据支撑"，绝不模糊带过
- 复杂数值计算（ROI测算/成本分配/漏斗预估等）用Python脚本完成
- 用户要求输出正式文档时可创建飞书文档
- 用户提供飞书文档链接（PRD/历史实验等）时读取作为上下文
