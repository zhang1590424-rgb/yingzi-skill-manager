# Skill 轻卡片式列表条目设计

日期：2026-05-22

## 1. 背景

当前技能列表和 Agent / 项目应用双栏已经具备基础可用性，但条目之间的视觉分组偏弱。Skill 数量增加后，用户需要更快扫出每个条目的名称、描述、状态、启用范围和异常提示。

本次调整目标不是把产品改成 marketplace，而是在本地工作台语境下增强列表条目的可读性和可操作性。

## 2. 核心判断

| 方案 | 判断 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 纯列表增强 | 不作为主方案 | 信息密度高，但条目边界仍弱 | Skill 多时仍需要较多视觉定位成本 |
| 轻卡片式列表行 | 采用 | 只把每一行变成浅底、细边框、小圆角的列表容器，不改变工作台结构 | 更容易区分条目、状态和操作对象，同时保持扫描效率 |
| 市场式大卡片 | 不采用 | 来源、下载量、安装按钮是 marketplace 语境，不是本地 Skill 管理核心 | 会让界面变成浏览市场，降低管理效率 |

## 3. 设计原则

- 允许“列表行卡片化”，禁止卡片网格和市场式大卡片。
- 卡片只是条目容器，不引入大阴影、头像墙、下载量、来源仓库等无关信息。
- 技能列表承担理解和选择，信息可以稍丰富。
- 应用双栏承担应用和收回，信息必须更紧凑。
- 状态必须文字 + 颜色共同表达，不只依赖颜色。
- 技能列表默认不选中 Skill，列表区域优先填满工作台主区域。
- 技能列表的详情只在用户点击卡片内容区后按需出现，用于确认使用情况、状态和正文缩略。
- 应用双栏保留选中态左侧短指示条，方便在移动 Skill 时定位当前位置。

## 4. 技能列表条目

技能列表条目用于回答：

- 这个 Skill 是什么？
- 是否有格式或分发问题？
- 已经在哪些地方启用？
- 是否值得被批量分发？

结构：

```text
[勾选]  Skill 名称                     [2 处启用] [1 个问题]
        描述，两行以内
        #tag #tag #tag                 hover: [使用] [打开] [删除]
```

交互：

| 状态 | 表现 |
|---|---|
| 默认 | 白色或近白浅底，细边框，小圆角 |
| Hover | 背景轻微加深，边框变清楚，并显示使用、打开、删除三个行内操作 |
| Selected | 仅在已打开详情时表达当前详情对象，不作为进入页面的默认状态 |
| Checked | 勾选按钮保持明确选中状态，不依赖整行变色 |
| 有问题 | 保留问题数量文字，使用低饱和警示色 |

操作规则：

| 操作 | 表现 | 为什么 |
|---|---|---|
| 使用 | Hover 后显示主操作按钮，打开原分发抽屉 | 常用动作靠近对象，减少移动距离 |
| 打开 | Hover 后显示图标按钮，打开本地 Skill 路径 | 排查和编辑文件时不必先进入详情 |
| 删除 | Hover 后显示风险按钮，继续沿用确认弹窗 | 删除是高风险动作，不能一键静默执行 |
| 点击卡片内容区 | 打开右侧详情侧栏 | 把详情变成确认步骤，不占用默认浏览空间 |

默认进入「技能列表」时，列表不自动选中第一个 Skill。这样用户看到的是完整 Skill 清单，而不是被系统替他决定要看哪一个 Skill。

## 5. 应用双栏条目

应用双栏条目用于回答：

- 当前 Skill 是已应用还是待应用？
- 状态是否异常？
- 项目聚合时启用了多少 Agent？

结构：

```text
Skill 名称                         [已启用]
简短描述 / skill name              [2/3 Agent]
```

交互：

| 状态 | 表现 |
|---|---|
| 默认 | 紧凑轻卡片，支持点击选中 |
| Hover | 明确可操作，边框和背景轻微变化 |
| Selected | 浅灰实底 + 左侧短指示条 |
| Dragging | 降低透明度，目标栏边框强化 |
| 异常 | 保留在已应用侧，状态徽标显示“冲突 / 未入库 / 路径不存在”等文字 |

应用双栏不展示标签、不提供复选框、不承载完整 `SKILL.md` 阅读，避免拖拽和双击操作被干扰。

## 6. 视觉约束

| 元素 | 规则 |
|---|---|
| 圆角 | 7-8px，保持桌面工具感 |
| 边框 | 默认使用浅边框，hover / selected 时增强 |
| 阴影 | 常规条目不用大阴影，最多使用极轻 1px 级层次 |
| 背景 | 使用中性浅灰与白色层级，不使用紫色渐变、发光、玻璃拟态 |
| 间距 | 技能列表条目间距略大于双栏条目；双栏保证一屏能容纳更多 Skill |
| 文本 | 名称单行省略，描述最多两行，标签最多展示前三个 |

## 7. 技能列表详情侧栏

技能列表详情侧栏只回答三个问题：

- 这个 Skill 当前能不能用？
- 它已经应用在哪些地方？
- `SKILL.md` 正文大概是什么内容？

结构：

```text
Skill 名称                         [已启用 2 处] [无问题]
描述

使用情况
  格式状态      SKILL.md 正常
  应用状态      Codex / 全局、项目 / Agent
  标签          #tag #tag

正文缩略
  SKILL.md 摘要，高度受控，可展开

技术信息
  默认折叠，展示主库路径
```

详情侧栏不再重复承担默认使用入口；使用、打开和删除已经前置到卡片 hover 操作。详情里的信息密度要服务确认，不变成完整文件阅读器。

## 8. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 技能列表条目可清楚区分名称、描述、标签、启用数和问题数。
- 进入技能列表默认不选中 Skill，列表区域填满主工作区。
- Hover 技能卡片时出现使用、打开、删除三个操作；移开后不占用扫描空间。
- 点击卡片非操作按钮区域后出现结构化详情侧栏。
- 全局 Agent / 项目应用双栏条目更像可移动对象，不像市场安装卡片。
- 选中、hover、拖拽、异常状态都清晰可辨。
- 常见桌面窗口尺寸下文字不重叠、不溢出，列表仍能独立滚动。
