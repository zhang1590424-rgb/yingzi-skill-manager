# Skill 轻卡片式列表条目设计

日期：2026-05-22

## 1. 背景

当前技能列表和 Agent / 项目应用双栏已经具备基础可用性，但条目之间的视觉分组偏弱。Skill 数量增加后，用户需要更快扫出每个条目的名称、描述、状态、启用范围和异常提示。

本次调整目标不是把产品改成 marketplace，而是在本地工作台语境下增强列表条目的可读性和可操作性。

## 2. 核心判断

| 方案 | 判断 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 纯列表增强 | 不作为主方案 | 信息密度高，但条目边界仍弱 | Skill 多时仍需要较多视觉定位成本 |
| 轻卡片式列表行 | 采用 | 只把每一行变成浅底、细边框、小圆角的列表容器，不改变工作台结构 | 更容易区分条目、状态和操作对象，同时保持扫描效率 |
| 市场式大卡片 | 不采用 | 来源、下载量、安装按钮是 marketplace 语境，不是本地 Skill 管理核心 | 会让界面变成浏览市场，降低管理效率 |

## 3. 设计原则

- 允许“列表行卡片化”，禁止卡片网格和市场式大卡片。
- 卡片只是条目容器，不引入大阴影、头像墙、下载量、来源仓库等无关信息。
- 技能列表承担理解和选择，信息可以稍丰富。
- 应用双栏承担应用和收回，信息必须更紧凑。
- 状态必须文字 + 颜色共同表达，不只依赖颜色。
- 选中态必须有左侧短指示条，方便在长列表中定位当前位置。

## 4. 技能列表条目

技能列表条目用于回答：

- 这个 Skill 是什么？
- 是否有格式或分发问题？
- 已经在哪些地方启用？
- 是否值得被批量分发？

结构：

```text
[勾选]  Skill 名称                     [2 处启用] [1 个问题]
        描述，两行以内
        #tag #tag #tag
```

交互：

| 状态 | 表现 |
|---|---|
| 默认 | 白色或近白浅底，细边框，小圆角 |
| Hover | 背景轻微加深，边框变清楚 |
| Selected | 浅灰实底，边框加深，左侧短指示条 |
| Checked | 勾选按钮保持明确选中状态，不依赖整行变色 |
| 有问题 | 保留问题数量文字，使用低饱和警示色 |

## 5. 应用双栏条目

应用双栏条目用于回答：

- 当前 Skill 是已应用还是待应用？
- 状态是否异常？
- 项目聚合时启用了多少 Agent？

结构：

```text
Skill 名称                         [已启用]
简短描述 / skill name              [2/3 Agent]
```

交互：

| 状态 | 表现 |
|---|---|
| 默认 | 紧凑轻卡片，支持点击选中 |
| Hover | 明确可操作，边框和背景轻微变化 |
| Selected | 浅灰实底 + 左侧短指示条 |
| Dragging | 降低透明度，目标栏边框强化 |
| 异常 | 保留在已应用侧，状态徽标显示“冲突 / 未入库 / 路径不存在”等文字 |

应用双栏不展示标签、不提供复选框、不承载完整 `SKILL.md` 阅读，避免拖拽和双击操作被干扰。

## 6. 视觉约束

| 元素 | 规则 |
|---|---|
| 圆角 | 7-8px，保持桌面工具感 |
| 边框 | 默认使用浅边框，hover / selected 时增强 |
| 阴影 | 常规条目不用大阴影，最多使用极轻 1px 级层次 |
| 背景 | 使用中性浅灰与白色层级，不使用紫色渐变、发光、玻璃拟态 |
| 间距 | 技能列表条目间距略大于双栏条目；双栏保证一屏能容纳更多 Skill |
| 文本 | 名称单行省略，描述最多两行，标签最多展示前三个 |

## 7. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 技能列表条目可清楚区分名称、描述、标签、启用数和问题数。
- 全局 Agent / 项目应用双栏条目更像可移动对象，不像市场安装卡片。
- 选中、hover、拖拽、异常状态都清晰可辨。
- 常见桌面窗口尺寸下文字不重叠、不溢出，列表仍能独立滚动。
