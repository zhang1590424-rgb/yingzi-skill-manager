# Skill 应用轻量看板样式设计

日期：2026-05-22

## 1. 背景

Agent 应用和项目应用的本质不是浏览详情，而是在当前对象上管理 Skill 是否生效。现有双栏已经具备基础能力，但视觉上更像普通列表，用户对“从一列移动到另一列”的感知不够强。

本轮只调整表达方式：把 Agent 应用和项目应用统一成两列轻量看板，不引入第三列，也不把异常状态做成用户需要日常理解的主状态。

## 2. 核心决策

| 维度 | 决策 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 主状态 | 只保留 `未应用 Skill` 和 `已应用 Skill` 两列 | 应用管理的核心问题只有“是否作用于当前对象” | 用户不需要理解额外状态分类 |
| 视觉隐喻 | 使用轻量看板列和卡片 | Skill 的应用/收回就是在两列之间移动对象 | 拖拽、双击、箭头操作更直觉 |
| 列顺序 | 左侧 `未应用 Skill`，右侧 `已应用 Skill` | 从左到右表达“应用到当前对象”的动作方向 | 应用是向右移动，收回是向左移动 |
| 异常表达 | 不做第三列；只在卡片或详情中轻量提示 | 冲突、未入库、路径问题来自本地文件系统，不应成为主导航概念 | 问题仍可被发现，但不会让主界面变复杂 |
| 项目聚合 | 项目应用仍按所有 Agent 聚合 | 第一版项目管理目标是便捷，不做项目内单 Agent 精细控制 | 用户一次操作即可影响项目下所有 Agent |

## 3. 工作区结构

```text
当前 Agent / 项目
┌──────────────────────────┬──────────────────────────┐
│ 未应用 Skill        数量  │ 已应用 Skill        数量  │
│ ┌ skill-card ┐            │ ┌ skill-card ┐            │
│ └────────────┘            │ └────────────┘            │
└──────────────────────────┴──────────────────────────┘
```

说明：

- `未应用 Skill`：当前对象尚未生效的主库 Skill。
- `已应用 Skill`：当前对象已经生效、部分生效，或占用了目标位置且需要用户确认的 Skill。
- 内部仍保留冲突、未入库、失效链接、路径不存在、格式异常等检测结果，但这些不升级为第三列。

## 4. 卡片表达

每张卡片优先展示：

1. Skill 显示名。
2. 描述或目录名。
3. 当前列语义：未应用 / 已应用。
4. 项目视图下的 Agent 覆盖数量，例如 `2/3 Agent`。
5. 如存在冲突或路径问题，在卡片底部用轻量提示说明，不用醒目的“异常状态列”。

技能组合不是第二套卡片。组合行必须复用同一张 Skill 卡片的基础样式，只增加展开控件和成员计数；展开后的成员行使用更轻的列表子行，不做卡片套卡片。

## 5. 操作规则

| 操作 | 结果 |
|---|---|
| 从未应用拖到已应用 | 应用 Skill |
| 从已应用拖到未应用 | 收回 Skill |
| 双击未应用卡片 | 应用 Skill |
| 双击已应用卡片 | 收回 Skill |
| 选中后点向右箭头 | 应用 Skill |
| 选中后点向左箭头 | 收回 Skill |
| 存在覆盖风险 | 操作前弹确认 |
| 当前无法直接处理 | 保持卡片所在列，详情或顶部提示给出下一步 |

## 6. 不做范围

- 不新增 `待处理`、`异常` 或类似第三列。
- 不改变主库优先、软链接分发和项目聚合规则。
- 不做项目内单 Agent 精细开关。
- 不新增操作日志或覆盖前备份。

## 7. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- Agent 应用和项目应用都显示为两列轻量看板。
- 应用方向为从左到右，收回方向为从右到左。
- 拖拽、双击、箭头按钮仍能完成应用 / 收回。
- 常见桌面窗口尺寸下卡片不重叠、不溢出。
