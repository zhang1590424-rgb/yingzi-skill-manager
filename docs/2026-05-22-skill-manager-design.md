# 技能中枢设计文档

日期：2026-05-22

## 1. 背景

本项目参考 `xingkongliang/skills-manager`，但不做通用开源工具克隆。参考项目覆盖 15+ coding tools，并包含 marketplace、presets、global workspace、project workspace、Git backup、activity log、CLI 等完整能力。

本项目面向个人本地使用，核心是把 Skill 作为知识资产统一管理，并清楚看到它们在 Trae、Codex、Claude Code 以及不同项目中的生效状态。

参考来源：

- `xingkongliang/skills-manager`：https://github.com/xingkongliang/skills-manager
- Impeccable 设计规范：https://impeccable.style/
- Impeccable product / brand 区分：https://impeccable.style/designing/
- Impeccable anti-pattern catalog：https://impeccable.style/slop/

## 2. 产品定位

产品中文名暂定为「技能中枢」。

一句话定义：

> 左侧选择管理范围，中间扫读技能和状态，右侧理解上下文并执行操作。

第一版目标：

- 做一个本地可视化桌面应用。
- 支持 Trae、Codex、Claude Code。
- 支持全局 Skill 和项目级 Skill 管理。
- 以 `~/.skills-manager` 作为统一主库。
- 通过软链接分发 Skill。
- 支持手动添加项目路径。

第一版不做：

- 通用 marketplace 浏览、搜索和推荐。
- 云同步。
- 多账号。
- 自动扫描所有项目。
- 覆盖前备份。
- 操作日志。
- 10+ Agent 的广泛适配。

## 3. 核心决策

| 维度 | 决策 |
|---|---|
| 产品形态 | 可视化桌面应用 |
| 技术方向 | Tauri 2 + React + TypeScript + SQLite |
| 支持 Agent | Trae / Codex / Claude Code |
| 主库路径 | `~/.skills-manager` |
| Skill 主体 | `~/.skills-manager/skills/<skill-name>` |
| 存量处理 | 导入主库，并保持原有 Agent / 项目可用 |
| 增量处理 | 直接进入主库，再通过桌面端分发 |
| 公司市场安装 | 从 `skills.bytedance.net` 技能详情链接入主库，不直接分发 |
| 分发方式 | 软链接 |
| 冲突策略 | 主库优先覆盖目标同名 Skill |
| 备份 / 日志 | 第一版不做 |
| 项目管理 | 手动添加项目路径 |
| UI 视角 | 技能主库、全局应用、项目应用、技能套装、设置 |

## 5. 本地目录结构

```text
~/.skills-manager/
  skills/
    code-review/
      SKILL.md
      ...
    product-research/
      SKILL.md
      ...
  app.db
  config.json
```

| 目录 / 文件 | 作用 |
|---|---|
| `skills/` | 技能主库，每个子目录是一个 Skill |
| `app.db` | 当前状态索引，记录 Skill、Agent、项目、分发关系 |
| `config.json` | 主库路径、Agent 路径、界面偏好 |

Skill 目录至少包含：

```text
skill-name/
  SKILL.md
```

可选包含：

```text
skill-name/
  README.md
  scripts/
  assets/
  references/
```

## 6. 数据模型

| 概念 | 记录内容 | 用户看到什么 |
|---|---|---|
| `skills` | 主库里的技能名称、路径、描述、标签 | 技能主库列表 |
| `agents` | Trae / Codex / Claude Code 的全局技能目录 | 全局应用页 |
| `projects` | 用户手动添加的项目路径 | 项目应用页 |
| `deployments` | 某技能是否分发到某 Agent / 某项目 | 启用状态 |
| `settings` | 主库路径、软链接策略、UI 偏好 | 设置页 |

`deployments` 只作为当前状态索引，不作为历史日志。

## 7. 核心流程

### 7.1 首次启动：导入存量技能

```text
检测 Agent 技能目录
→ 展示发现结果
→ 标记：可入库 / 同名覆盖 / 疑似技能
→ 用户确认
→ 入主库并重建软链接
```

处理规则：

| 类型 | 处理 |
|---|---|
| 主库没有，Agent 里有 | 导入主库，然后把原目录替换成指向主库的软链接 |
| 主库已有，Agent 里同名 | 以主库为准，目标目录替换为软链接 |
| 目标本来就是软链接 | 校验是否指向主库，不是则重建 |
| 识别不到 `SKILL.md` | 显示为疑似技能，不自动处理 |

### 7.2 分发技能

```text
选择技能
→ 点击「分发」
→ 右侧抽屉选择目标：Trae / Codex / Claude Code / 项目
→ 展示将创建的软链接路径
→ 确认分发
```

分发结果示例：

```text
~/.codex/skills/code-review -> ~/.skills-manager/skills/code-review
```

项目级示例：

```text
<project>/.codex/skills/code-review -> ~/.skills-manager/skills/code-review
```

### 7.3 收回技能

收回只删除目标目录中的软链接，不删除主库技能。

| 场景 | 处理 |
|---|---|
| 目标是指向主库的软链接 | 删除软链接 |
| 目标是普通目录 | 不自动删除，显示为非本应用分发 |
| 主库技能本身 | 不受影响 |

### 7.4 处理冲突

状态为「冲突」时，右侧详情显示：

- 主库路径。
- 目标路径。
- 当前目标类型：普通目录 / 指向别处的软链接。
- 操作：用主库覆盖。

不提供复杂合并流程。第一版按主库优先。

### 7.5 删除技能

| 情况 | 处理 |
|---|---|
| 未分发 | 允许删除主库目录 |
| 已分发 | 提示先收回所有分发 |
| 用户强制删除 | 第一版不做 |

### 7.6 导入技能

导入技能统一处理外部来源入主库，包含本地技能文件夹和公司 Skill 市场详情链接。目标不是在本产品里重做 Skill 市场，而是把公司内部市场作为一个可信入库来源。

```text
点击「导入技能」
→ 选择「本地文件夹」或「市场链接」
→ 本地文件夹：选择包含 SKILL.md 的目录并复制到主库
→ 市场链接：粘贴技能市场详情链接
→ 解析 source / skill / version
→ 调用官方 skills 安装器安装到临时项目
→ 从临时项目复制 Skill 到主库
→ 刷新主库列表
```

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 只支持技能详情链接 | 链接里能稳定解析来源仓库和 Skill 名称，避免依赖页面视觉结构 | 老张从市场复制链接即可安装，不需要理解仓库地址 |
| 使用官方 `skills` 安装器 | 内部市场可能有鉴权、版本和包结构规则，自己抓页面容易失真 | 与市场官方安装结果一致，失败原因也更接近真实环境 |
| 安装到临时项目后再复制入主库 | 官方安装器默认面向 Agent / 项目，本产品的事实源是主库 | 不会绕过主库，也不会直接污染全局 Agent 目录 |
| 不自动分发到 Agent | 入库和启用是两个不同决策 | 安装后仍需手动选择分发范围，降低误启用风险 |
| 不提供「新建技能」主入口 | 当前最常见任务是汇聚和分发现有 Skill，空白创建容易和导入混淆 | 顶部操作更少，老张只需要判断“导入后是否分发” |

约束：

- 需要能访问 `skills.bytedance.net` 和 `bnpm.byted.org`。
- 需要本机 Node / npx 环境可用。
- 如内部 SSO 或 CLI 鉴权失败，界面展示原始错误摘要，不伪造安装成功。

## 8. 状态定义

| 状态 | 含义 |
|---|---|
| 已启用 | 目标目录存在指向主库的软链接 |
| 未启用 | 目标目录没有该技能 |
| 未入库 | 目标目录有技能，但主库没有 |
| 冲突 | 目标目录有同名普通目录或指向别处的软链接 |
| 失效 | 软链接目标不存在 |
| 路径不存在 | Agent 或项目技能目录不存在 |

状态展示必须使用文字 + 徽标，不只依赖颜色。

## 9. UI / UX 设计

### 9.1 设计原则

采用「Codex app 式中文本地工作台」：

- 三栏结构。
- 列表优先。
- 状态优先。
- 少弹窗。
- 少卡片。
- 无装饰型 AI 风格。
- 所有高风险操作都在上下文里确认。

结合 Impeccable 的 product mode 原则：

- 设计服务任务，不做品牌展示。
- 先明确用户、场景、反例，再做界面。
- 迭代后要做可访问性、响应式、异常状态和文案打磨。

### 9.2 主界面

```text
┌────────────────────────────────────────────────────────┐
│ 技能中枢        搜索技能、项目、智能体应用       刷新  设置 │
├──────────┬────────────────────────┬────────────────────┤
│ 主库      │ 技能列表                │ 当前技能上下文        │
│ 全局      │ 名称 / 状态 / 启用范围    │ 摘要                 │
│ 项目      │ 标签 / 问题提示          │ SKILL.md 预览        │
│ 套装      │                        │ 分发位置             │
│ 设置      │                        │ 操作                 │
└──────────┴────────────────────────┴────────────────────┘
```

| 区域 | UX 目标 |
|---|---|
| 左栏 | 切换管理范围，不承载复杂信息 |
| 中栏 | 快速扫读所有技能和状态 |
| 右栏 | 解释当前对象，并给出下一步动作 |
| 顶部搜索 | 搜技能、项目、Agent、路径 |

### 9.3 页面设计

#### 技能主库

默认首页。

核心内容：

- 技能列表。
- 搜索和标签筛选。
- 技能详情。
- `SKILL.md` 预览。
- 启用位置。
- 分发 / 收回 / 打开目录。

#### 全局应用

按 Agent 查看全局技能状态。

```text
全局应用
[Trae] [Codex] [Claude Code]

code-review        已启用
product-research   未启用
local-only-skill    未入库
old-skill           冲突
```

#### 项目应用

只显示用户手动添加过的项目。

核心操作：

- 添加项目。
- 移除项目。
- 给项目启用技能。
- 停用项目技能。
- 查看某项目下 Trae / Codex / Claude Code 的 Skill 状态。

#### 技能套装

中文化的 preset。

示例：

| 套装 | 包含技能 | 用途 |
|---|---|---|
| 产品经理常用 | 调研、PRD、竞品分析 | 知识管理 / 产品设计 |
| 开发协作 | code-review、unit-test、debug | 开发项目 |
| 收尾归档 | neat-freak、文档同步 | 会话结束整理 |

#### 设置

设置独立成页，不塞进复杂弹窗。

内容：

- 主库路径。
- Trae / Codex / Claude Code 全局技能目录。
- 项目列表。
- 分发策略：固定软链接。
- 冲突策略：固定主库优先。

### 9.4 视觉系统

| 元素 | 规则 |
|---|---|
| 背景 | 中性浅灰，接近桌面应用工作区；侧栏和顶部栏允许低强度半透明材质，不用偏暖纸张色或刺眼纯白 |
| 主色 | 只用于选中态、焦点态和主操作，避免把整页染成品牌色 |
| 状态色 | 绿色=已启用，橙色=冲突，红色=失效，灰色=未启用 |
| 字体 | 中文使用系统字体，路径和代码使用等宽字体 |
| 圆角 | 小圆角，偏工具感 |
| 阴影 | 只用于浮层、抽屉和批量操作条；常规结构用分割线、留白和轻微底色表达层级 |
| 图标 | 线性图标，辅助识别，不代替文字 |

### 9.4.1 应用图标

桌面图标采用「抽象网」方向，不画蜘蛛本体。

| 元素 | 规则 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 底色 | 白色 macOS 圆角方形底座，保留轻微阴影 | 与常用 Dock 图标家族协调，降低工具感里的压迫感 | 长时间停留在 Dock 中不刺眼，识别稳定 |
| 主符号 | 深石墨色三段式开放网形 | 用最少线条表达连接、编织和分发，不陷入复杂蜘蛛网 | 小尺寸下仍能看清，不需要解释功能细节 |
| 强调色 | 中央一个青绿色圆点 | 表达本地 Skill 主库 / 唯一事实源 | 一眼知道“中心”在哪里 |
| 禁止项 | 不用文字、蜘蛛、复杂网格、紫色渐变、发光和 AI 装饰 | 避免变成营销图标或爬虫/监控意象 | 品牌更安静、高级，贴近本地工作台 |

### 9.4.2 Codex app 式工作台细则

目标不是复刻某个具体皮肤，而是复刻 Codex app 的产品气质：安静、可信、上下文连续、可长时间工作。

| 维度 | 规则 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 信息密度 | 三栏固定，列表行高控制在 54-64px，详情区用分组而不是大卡片 | Skill 管理是反复扫读和判断状态，不是浏览营销内容 | 老张能更快看出哪个技能启用、冲突或需要处理 |
| 色彩 | 使用冷中性灰作为底色，主操作用近黑或深灰，状态色低饱和 | 本地工具需要稳定感，过暖或高饱和会让状态判断变累 | 长时间使用不刺眼，风险状态仍然明显 |
| 层级 | 左侧导航最安静，中间列表最可扫，右侧详情最明确 | 三栏各司其职，避免所有区域都抢注意力 | 当前对象和下一步动作更清楚 |
| 控件 | 按钮、搜索框、分段控件统一 7-8px 圆角，边框细、背景轻 | Codex app 的精致感来自一致控件尺度，而非装饰 | 新建、导入、分发等操作更像同一套系统 |
| 选中态 | 选中项用浅底色 + 细边框 + 左侧短指示，不只靠文字加粗 | 状态要一眼可定位，也要兼顾色弱用户 | 在长列表中不容易丢失当前位置 |
| 细节 | hover / focus / disabled 都要有状态；焦点环不移除 | 桌面工具大量依赖键盘和鼠标 | 键盘操作和低视力用户都能安全使用 |
| 动效 | 只保留 150-220ms 的状态过渡，尊重 reduced-motion | 动效用于反馈，不用于装饰 | 操作更顺，不产生“AI 风”噪音 |

半透明使用边界：

- 只用于左侧导航、顶部工具栏、选中胶囊、抽屉遮罩等“系统材质”区域。
- 透明度要低，必须保留足够文字对比；正文内容区、列表内容和详情阅读区保持实底。
- 不使用发光边框、强 blur、彩色渐变玻璃、卡片套卡片。
- 背景可有非常轻的冷暖色漂移，用来模拟 macOS 窗口材质，但不能成为视觉主角。

### 9.5 组件规则

| 组件 | 设计方式 |
|---|---|
| 技能列表 | 表格 / 紧凑列表，不用卡片网格 |
| 状态徽标 | 文字 + 小圆点 |
| 操作按钮 | 每屏只保留一个主按钮 |
| 分发面板 | 右侧抽屉，不用大弹窗 |
| 高风险确认 | 只有主库覆盖目标等操作才弹确认 |
| 空状态 | 一句话说明 + 一个动作 |
| 路径展示 | 等宽字体，可复制，可打开目录 |

### 9.6 明确避免的 UI 反模式

- 紫色渐变。
- 玻璃拟态。
- 发光边框。
- 卡片套卡片。
- 所有按钮都像主按钮。
- 复杂设置塞弹窗。
- 大段说明文。
- 灰字低对比。
- 无意义动效。
- 为了显得技术化而滥用等宽字体。

### 9.7 体验修复约束

以下约束用于修复首轮 UI / UX 审视发现的问题，后续实现必须保持一致：

| 问题 | 规则 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 跨页面详情串页 | 每个管理视图只能展示本视图对象；切换视图、Agent、项目时同步更新右侧详情选择 | 三栏工作台的核心是上下文连续，不能让旧对象污染新视图 | 老张切换范围后，右侧动作始终作用在当前列表对象上 |
| 高风险删除 | 删除 Skill / 套装必须先进入确认弹窗；已分发 Skill 不允许直接删除 | 主库是唯一事实源，误删成本高 | 降低误删和“以为只是停用”的风险 |
| 分发确认 | 分发抽屉必须展示最终软链接路径、目标当前状态和冲突风险；只有存在真实覆盖风险时才强化警告 | “主库优先”是策略，不是每次操作都同等危险 | 老张确认前能知道会写到哪里、覆盖什么 |
| 弹窗 / 抽屉 | 弹窗和抽屉使用 `role="dialog"`、`aria-modal`、可 Esc 关闭、焦点进入浮层 | 桌面工具需要可键盘操作和读屏可理解 | 不依赖鼠标也能完成导入、分发和确认 |
| 字号与密度 | 工作台正文以 13px 为主，列表说明和路径以 11.5-12px 为主，标题控制在 16-17px | 这是高频管理工具，不是营销页面；字号过大降低信息密度 | 界面更精致，列表能扫更多信息且不显笨重 |
| 点击面积 | 关键按钮命中区不小于 38-40px；图标按钮可视觉克制但命中区要足够 | 小控件密集时容易误点 | 保持精致外观，同时提高操作容错 |
| 小窗口策略 | 桌面端最小宽度按 920px 设计，不在更窄宽度硬挤三栏 | 三栏产品需要稳定的信息结构 | 避免窄窗口下文字重叠或动作区被挤压 |

### 9.8 首轮诊断问题修复验收

本轮修复目标是把诊断出的 UI / UX 问题全部收敛到可验证状态：

| 诊断问题 | 修复验收 |
|---|---|
| `SKILL.md` frontmatter 被当成正文展示，`description: |` 被解析成孤立竖线 | 技能名称、描述和标签按 frontmatter 正确解析；预览默认只展示去掉 frontmatter 后的正文摘要，不把元数据或大段系统提示顶到详情主区域 |
| 项目应用无项目时仍展示旧技能详情和操作 | 当没有项目或当前项目失效时，中栏和右栏都进入项目空状态，只保留“添加项目”动作，不出现主库技能的分发 / 删除操作 |
| 分发抽屉让用户阅读技术路径来判断风险 | 每个目标先展示人能理解的结果、当前状态和风险级别；完整软链接路径作为二级信息；只有冲突 / 未入库 / 异常时强化警告 |
| 技能列表不利于扫读和筛选 | 主库列表提供状态筛选：全部、有问题、已启用、未启用；列表行展示标签、问题数、启用数，长描述可换行两行 |
| 视觉偏玻璃拟态 | 主体结构改为实底中性表面，blur 只保留在浮层遮罩；不使用装饰性渐变和发光 |
| 控件点击面积偏小 | 常用按钮、图标按钮、勾选控件和分段按钮命中区达到 40-44px，视觉密度仍保持桌面工作台风格 |
| 小窗口硬锁 1040px | 桌面端按 920px 最小可用宽度设计，低于常规宽度时压缩侧栏和详情栏，不让内容重叠 |
| 导入市场链接缺少即时反馈 | 市场链接在提交前做基础格式校验，错误就近展示，并说明只支持公司 Skill 市场详情链接 |

### 9.9 易用性迭代：健康总览、问题修复、状态筛选

基本可用后，下一阶段优先提升“判断速度”和“操作确定感”，不扩大产品边界。

| 方向 | 规则 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 健康总览 | 在现有工作台内汇总冲突、失效链接、路径不存在、未入库、格式异常；问题项必须能跳到对应对象 | Skill 管理的核心不是浏览，而是发现需要处理的异常 | 老张打开应用后先知道哪里需要管，不必逐页排查 |
| 状态筛选 | 技能主库和应用状态列表提供全部、有问题、未入库、冲突、失效、未启用、已启用等筛选 | Skill 数量增加后，搜索只能找名称，不能回答“哪些需要处理” | 更快定位待修复对象和未启用技能 |
| 操作预检 | 分发前展示目标数量、覆盖风险、异常路径和最终软链接路径；明确说明不会修改主库内容 | 分发会写入 Agent / 项目目录，风险来自目标位置而不是主库 | 用户确认前知道会影响哪里、是否覆盖 |
| 操作回执 | 导入、分发、收回后展示成功、跳过、冲突、错误数量；错误文案需要包含下一步 | 只显示“成功/失败”不能帮助恢复 | 失败后能知道该去设置路径、处理冲突或重试 |
| 导入后引导 | 导入成功后选中新入库技能，右侧主操作保持“分发” | 入库和启用是两个决策，但下一步通常是分发 | 用户不需要重新搜索刚导入的技能 |

边界：

- 不做自动扫描所有项目。
- 不做通用市场浏览。
- 不做云同步、多账号或完整操作日志。
- 不改变主库优先和软链接分发策略。

## 10. 错误处理

| 错误 | 处理 |
|---|---|
| Agent 路径不存在 | 在设置页提示配置路径 |
| 项目路径不存在 | 项目应用页标记路径不存在 |
| 软链接创建失败 | 显示目标路径和失败原因 |
| 目标同名冲突 | 显示冲突状态，提供用主库覆盖 |
| 主库 Skill 缺少 `SKILL.md` | 标记为格式异常 |
| 软链接目标不存在 | 标记为失效，提供清理 |
| 权限不足 | 显示需要手动授权或调整目录权限 |

## 11. 验证标准

进入实现后，每次迭代需要验证：

- 桌面应用能启动。
- 三栏主界面正常展示。
- 技能主库能扫描 `~/.skills-manager/skills`。
- Trae / Codex / Claude Code 的全局路径能配置和刷新。
- 手动添加项目能显示项目级状态。
- 分发能创建软链接。
- 收回只删除指向主库的软链接。
- 同名冲突按主库优先处理。
- UI 没有文字重叠、按钮溢出、低对比状态。
- 常见桌面窗口尺寸下可用。

## 12. 后续实现步骤

设计确认后，进入实现阶段前应先寻找适合的技能：
