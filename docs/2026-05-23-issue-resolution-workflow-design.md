# 初始化后问题处理工作流设计

日期：2026-05-23

## 1. 背景

初始化结束后，健康总览可能发现失效链接等问题。只展示错误会把判断成本留给用户：用户不知道能不能忽略，也不知道处理会改动什么。

本次目标是把健康总览从“报警器”改成“收尾工具”：每类问题都必须给用户明确动作。

## 2. 核心决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 健康总览点击打开问题处理抽屉 | 问题需要解释、确认和批量处理，不适合直接跳筛选 | 用户先看清范围，再决定 |
| 每个问题至少提供忽略和处理 | 用户的真实选择只有两类：接受现状，或修复现状 | 不会被错误提示卡住 |
| 忽略后不再进入健康提示 | 忽略是明确决策，不应反复打扰 | 工作台保持安静 |
| 处理动作按问题类型定义 | 不同问题的修复方式不同，不能用一个模糊按钮 | 用户能预期系统会做什么 |

## 3. 第一版范围

第一版先处理当前真实出现的「失效链接」：

- 忽略：记录问题指纹，之后健康总览和问题筛选不再提示同一条失效链接。
- 处理：删除断裂软链接；删除后重新扫描，问题消失。

问题指纹由问题类型、目标范围、Agent、项目、Skill 名称、目标路径和链接目标共同组成。路径或链接目标变化时，会被视为新问题。

## 4. 交互设计

点击健康总览中的「失效链接」后打开右侧抽屉：

- 标题：`处理失效链接`
- 描述：`这些链接指向的 Skill 已经不存在。可以忽略它们，也可以删除这些失效链接。`
- 列表项展示 Skill 名称、Agent / 项目、原因和路径。
- 顶部批量动作：
  - `忽略全部`
  - `处理全部`
- 单项动作：
  - `忽略`
  - `处理`

处理前需要确认，因为会删除本地软链接；忽略不删除文件。

## 5. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- `npm run tauri:build`
- 点击健康总览能打开问题处理抽屉。
- 对失效链接执行「忽略全部」后，健康总览不再提示同一批问题。
- 对失效链接执行「处理全部」后，断裂软链接被删除，重新扫描后问题消失。
