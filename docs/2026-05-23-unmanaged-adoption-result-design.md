# 存量 Skill 入库结果修正设计

日期：2026-05-23

## 1. 背景

初始化流程已经提供「已有 Skill 导入」，但用户选择导入后，工作台仍可能出现「未入库」警告。当前真实案例中，Codex 全局 Skill 目录里的条目不是完整的存量 Skill，而是指向主库路径的断裂软链接：

```text
/Users/bytedance/.codex/skills/lark-base -> /Users/bytedance/.skills-manager/skills/lark-base
```

主库中对应目录不存在时，这类条目已经没有可复制的源内容，不能被当作“可入库的存量 Skill”。

## 2. 核心决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 断裂软链接归类为失效，不归类为未入库 | 未入库意味着有一份可导入的实体；断链只有一个失效引用 | 用户看到的问题更准确，不会误以为点击导入就能修复 |
| 初始化批量入库失败时不自动进入下一步 | 用户已经选择处理，失败后继续前进会制造“处理了但没生效”的错觉 | 用户能在初始化当场看到失败原因 |
| 入库后刷新存量列表 | 入库结果必须以重新扫描后的真实状态为准 | 已解决项从初始化列表消失，剩余项不被错误保留 |
| 工作台导入当前范围只处理真正未入库项 | 断链、冲突、格式异常需要不同处理方式 | 工作台按钮语义更稳定 |

## 3. 行为规则

- 扫描 Agent / 项目目录时，若发现未被主库匹配的软链接：
  - 软链接目标不存在：状态为 `broken`，文案为「软链接目标不存在」。
  - 软链接目标存在且包含 `SKILL.md`：状态为 `unmanaged`，允许入库。
  - 软链接目标存在但不含 `SKILL.md`：状态为 `invalid`，不进入入库流程。
- 初始化「已有 Skill 导入」只列出 `unmanaged` 状态。
- 批量入库返回错误时：
  - 保持在当前步骤。
  - 展示前 3 条失败原因。
  - 重新扫描未入库列表。
  - 不调用完成初始化。

## 4. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 断裂软链接不再显示为「未入库」。
- 初始化导入失败时不直接进入下一步。
- 已成功入库的 Skill 重新扫描后不再出现在未入库列表。
