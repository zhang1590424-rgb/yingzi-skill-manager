# 技能列表详情与首页噪音收敛设计

日期：2026-05-23

## 1. 背景

当前「技能列表」已经是主要工作入口，但存在三个影响主线的问题：

- 技能行的「使用 / 打开 / 删除」以绝对定位悬浮在行内容上，hover 后会遮挡描述和启用状态。
- 用户点击 Skill 打开右侧详情后，只能通过点击其他列表项或切换页面收回详情。
- 首页顶部的「健康总览」虽然有价值，但不是用户进入工作台后的主任务，视觉权重过高。

本次调整目标是降低非主线信息和操作噪音，让列表浏览、选择 Skill、查看详情这三件事更顺。

## 2. 核心判断

| 方案 | 判断 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 保留悬浮按钮，只调阴影和位置 | 不采用 | 遮挡本质来自绝对定位，视觉轻一点仍会压内容 | 看起来略好，但扫列表时仍突兀 |
| 把所有操作常驻展示 | 不采用 | Skill 多时每行都会出现一排按钮，管理界面会变成操作墙 | 可发现性更高，但扫描效率下降 |
| 使用稳定右侧操作区，hover 时替换状态信息 | 采用 | 操作仍靠近对象，但不覆盖名称、描述和标签 | 用户能快速扫内容，需要操作时也不用移动太远 |

## 3. 设计规则

- 技能列表行内操作不得使用覆盖内容的绝对定位浮层。
- 行右侧可以预留稳定区域：默认展示启用数 / 问题数，hover 或 focus 时切换为操作。
- 「使用」保留文字，因为它是对象主操作；「打开」「删除」使用图标按钮，配 `aria-label` 和 `title`。
- 右侧 Skill 详情顶部增加轻量关闭按钮，点击后收回详情侧栏并回到列表优先状态。
- 首页不再常驻展示健康总览；异常仍通过列表筛选、状态徽标、Agent / 项目应用页的具体对象表达。

## 4. 验证标准

- 技能行 hover 后操作按钮不遮挡名称、描述、标签和启用状态。
- 点击 Skill 后右侧详情出现，点击右上角关闭按钮后详情侧栏收回。
- 进入技能列表时顶部不再出现健康总览提示条。
- 技能组合、Agent 应用、项目应用、设置页不出现同类悬浮遮挡问题。
- `npm run build` 通过。
- `cargo check --manifest-path src-tauri/Cargo.toml` 通过。
