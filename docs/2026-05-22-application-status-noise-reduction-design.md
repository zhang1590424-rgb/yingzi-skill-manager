# 应用状态降噪设计

日期：2026-05-22

## 1. 背景

Agent / 项目应用页已经用双栏表达核心任务：左侧未应用，右侧已应用。当前界面又在顶部展示全部、未应用、已应用、需确认、未入库、冲突、失效等筛选，并在每个未应用 Skill 下重复展示“目标目录尚未创建，应用时会自动创建”。

问题本质是把系统内部可自愈状态当成用户问题暴露了。目标目录不存在但系统能自动创建，这不需要用户理解、确认或修复，应该静默处理。

## 2. 方案比较

| 方案 | 做法 | 取舍 |
|---|---|---|
| 保留全部筛选，只改文案 | 把“目标目录尚未创建”改成更轻的说明 | 仍然让用户看到不需要处理的信息，噪声没有消失 |
| 隐藏可自愈提示，保留顶部筛选 | 移除黄色标签，但顶部仍展示多个 0 状态 | 列表干净一些，但上方仍有重复和表意不明的问题 |
| 推荐方案：只显示可行动状态 | 自动创建目录完全静默；应用页顶部只在存在真实问题时显示非零问题筛选 | 页面默认更安静，用户只被需要处理的状态打断 |

## 3. 核心决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 可自动创建的目标目录不显示为黄色标签 | 这不是用户风险，也不需要用户采取动作 | 未应用列表更像任务清单，不再每行重复系统细节 |
| 可自动创建的目录不计入问题筛选和健康总览 | “问题”应只表示需要用户确认、覆盖、入库或修复 | 用户看到问题数时可以相信它真的需要处理 |
| 应用页移除全部、未应用、已应用常驻筛选 | 双栏标题和数量已经表达这些状态 | 顶部少一排重复标签，视觉重心回到应用 / 收回 |
| 普通行不再重复显示未应用 / 已应用徽标 | 行所在列已经给出状态，重复徽标只增加扫读负担 | 用户更容易看到名称、描述和真正异常 |
| 问题筛选只展示非零项 | 0 个冲突、0 个未入库没有决策价值 | 有问题才出现入口，没有问题时页面保持安静 |
| 分发预检中可自动创建目录按正常目标处理 | 预检要帮助判断风险，不是解释内部创建目录 | 只有会覆盖或阻断的位置被突出 |

## 4. 应用页状态表达

默认状态：

```text
标题区
双栏：未应用 Skill / 已应用 Skill
```

存在真实问题时：

```text
问题筛选：全部问题 3  冲突 1  未入库 2
双栏：仅展示当前筛选命中的条目
```

真实问题范围：

- 冲突：目标同名内容会被覆盖，需要确认。
- 未入库：目标已有主库外 Skill，需要决定是否导入。
- 失效：软链接目标不存在，需要重建或清理。
- 格式异常：缺少 `SKILL.md` 等不可正常应用。
- 项目根路径不可访问：会阻断写入，需要到设置修复。

不作为用户问题展示：

- Agent 全局 Skill 根目录尚未创建。
- 项目内某个 Agent 的 Skill 目录尚未创建，但项目根路径存在。
- 应用时可以直接创建的目标目录。

## 5. 验证标准

- Agent / 项目应用页未应用列表不再出现“目标目录尚未创建，应用时会自动创建”标签。
- 未应用列和已应用列中的普通行不再重复显示“未应用 / 已应用”徽标。
- 全局 Agent 页和项目应用页共用同一规则。
- 无真实问题时，应用页顶部不显示状态筛选条。
- 有真实问题时，只显示非零问题筛选，不显示未应用、已应用、全部等重复标签。
- 分发抽屉中可自动创建目录按普通可应用目标展示，项目根路径不可访问仍提示并阻断。
- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- `npm run tauri:build`
