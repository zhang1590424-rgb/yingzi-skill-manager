# 错误提示承载位置优化设计

日期：2026-05-23

## 1. 背景

导入技能失败时，错误提示显示在主页面顶部，但导入弹窗仍在前景。用户正在操作弹窗，却需要到遮罩背后找错误原因，反馈位置和操作上下文断开。

## 2. 设计决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 区分页内错误和浮层错误 | 错误应该出现在触发它的操作附近 | 用户不用在遮罩背后找失败原因 |
| 弹窗和抽屉错误使用悬浮 toast | 导入、组合保存、确认删除、分发都发生在浮层里，但内嵌错误条会改变浮层高度 | 操作失败后浮层保留，错误直接可见且不挤压内容 |
| 页面级操作也使用 toast | 刷新、设置页维护、普通页面按钮没有局部容器，插入顶部会造成内容跳变 | 主工作台布局保持稳定 |
| 错误文案去掉泛化排查建议 | “检查输入内容、目标路径和目录权限”太宽，不能指导下一步 | 用户先看到具体原因，必要时再看路径或设置 |

## 3. 交互规格

```text
页面操作失败
→ 窗口右上显示轻 toast

浮层操作失败
→ 窗口右上显示轻 toast，层级高于遮罩
→ 浮层保持打开
→ 用户修正后可直接重试
→ toast 很快自动消失，不要求用户手动关闭
```

适用浮层：

- 导入技能弹窗。
- 新建 / 编辑技能组合弹窗。
- 删除 / 移除确认弹窗。
- 分发抽屉。

## 4. 文案规则

- 优先保留具体失败原因。
- 删除 `下一步：检查输入内容、目标路径和目录权限后重试` 这类泛化尾巴。
- 同名冲突使用 `已存在同名技能：<name>`。
- 缺少 `SKILL.md` 使用 `这个文件夹缺少 SKILL.md` 或 `压缩包里没有找到 SKILL.md`。
- 路径阻断使用 `项目路径不存在，请先到设置修正`。

## 5. 验收标准

- 导入重复 Skill 时，错误显示为悬浮 toast，不在弹窗内撑高布局。
- 组合保存、确认弹窗、分发抽屉内的失败不再写到遮罩后的页面顶部。
- 页面级加载和普通页面操作也使用悬浮 toast。
- 错误文案短且可读，不包含泛化排查长句。
- toast 自动消失，不需要用户手动关闭。
- `npm run build` 通过。
