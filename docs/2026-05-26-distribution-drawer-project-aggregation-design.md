# 分发抽屉项目目标聚合设计

日期：2026-05-26

## 1. 背景

「项目应用」工作台已经按项目聚合：用户选择一个项目后，应用或收回 Skill 会影响该项目下所有已启用 Agent。

当前「技能列表」和「技能组合」里的分发抽屉仍把项目目标拆成「项目 / Agent」行，例如「工作 / Codex」。这和项目工作台的产品口径不一致，也把项目内单 Agent 精细选择暴露给用户。

## 2. 决策

| 维度 | 决策 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 项目目标展示 | 分发抽屉只展示项目行，不展示「项目 / Agent」行 | 项目级管理的核心是一次覆盖项目下所有 Agent | 用户选择项目即可完成全 Agent 分发，不需要重复理解每个 Agent 目录 |
| 真实执行目标 | 选择项目行后，前端展开为该项目下所有已启用 Agent 的 `DeployTarget` | 复用现有 Tauri 分发命令，不引入新后端语义 | 行为和项目工作台的应用 / 收回保持一致 |
| 预检统计 | 预检按展开后的真实目标数量和状态统计 | 风险应该按实际会写入的位置计算 | 冲突、路径异常和缺失成员数量更接近真实影响范围 |
| 路径展示 | 项目行使用聚合路径占位：`<项目>/{agent-skill-root}/{skill-name}` | 不再把项目内 Agent 作为可选项，但保留技术可解释性 | 用户知道会写入多个 Agent 目录，又不会被迫逐个选择 |

## 3. 交互规则

- 全局目标仍按 Agent 展示，例如「Codex / 全局」。
- 项目目标按项目展示，例如「工作 / 项目」。
- 选择一个项目目标后，确认分发会创建或更新该项目下所有已启用 Agent 的软链接。
- 应用技能组合时，组合本身仍不软链接，只处理组合内仍存在的成员 Skill。
- 项目路径不存在时，项目行禁用确认分发，并提示先到设置修复。
- Agent / 项目下的 Skill 根目录不存在时，仍视为可创建目录，不作为阻断风险。

## 4. 验证

- 在技能列表点击「使用」，抽屉中项目只出现项目行。
- 在技能组合点击「应用组合」，抽屉中项目只出现项目行。
- 选择项目行后，预检目标数量等于该项目下已启用 Agent 数量乘以技能数量。
- 确认后创建的软链接覆盖该项目下所有已启用 Agent。
