# 初始化引导与边界状态设计

日期：2026-05-22

## 1. 背景

当前应用已经具备技能列表、技能组合、Agent / 项目应用、设置和分发预检，但首次打开时仍容易遇到三个空白判断：

- 主库里没有技能时，用户不知道先导入还是先看 Agent。
- Agent 或项目路径不存在时，用户不容易区分“可自动创建的目录”和“必须先修复的项目根路径”。
- 组合为空、成员缺失、加载失败等边界状态已有底层保护，但界面上的下一步不够集中。

本轮目标不是做教学流程，而是在真实工作台里提供初始化引导和可恢复的边界状态。

## 2. 核心决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 使用非阻塞初始化清单 | 本地工具的首要价值是尽快看到真实 Skill 状态，不应该先进入教程 | 用户可以按清单完成导入、确认 Agent、添加项目，也可以直接跳过 |
| 空状态直接给下一步 | 空白不是错误，应该告诉用户当前区域会承载什么和下一步动作 | 没有技能、没有组合、没有项目时不会像应用坏了 |
| 区分可创建路径与阻断路径 | Agent Skill 根目录不存在可以在应用时创建，项目根目录不存在会导致误写入风险 | 用户知道哪些问题可以直接继续，哪些必须先去设置修复 |
| 加载失败保留恢复入口 | 本地文件和 SQLite 可能因为权限、路径、占用而失败 | 用户能刷新重试、看到配置目录，不会卡在空白屏 |
| 组合重名创建不静默覆盖 | 新建组合与编辑组合是不同意图，重名覆盖会造成配置丢失 | 用户创建同名组合时得到明确错误，已有组合不会被误改 |

## 3. 初始化引导

引导以顶部轻量面板展示，满足以下条件时出现：

- 用户尚未关闭本轮引导。
- 以下任务尚未全部完成：
  - 主库至少有 1 个正式 Skill，或发现可入库的存量 Skill。
  - 至少有 1 个 Agent 处于可管理状态。
  - 至少有 1 个 Skill 已应用到任意 Agent 或项目。
  - 至少有 1 个项目被手动纳入管理（可选任务）。

引导不阻塞任何操作，允许关闭；关闭状态存入 `localStorage`。

## 4. 边界状态

| 场景 | 界面处理 |
|---|---|
| 主库为空、无存量 Skill | 技能列表空状态显示“导入技能”，右侧说明导入后不会自动分发 |
| 主库为空、发现全局存量 Skill | 右侧显示“导入全部全局存量”，减少逐个处理成本 |
| 搜索无结果 | 保留筛选状态，提示没有匹配项，不建议用户新建无关对象 |
| 项目列表为空 | 项目应用区和设置页都提供“添加项目”，但项目不是首次启动必填项 |
| Agent 全局目录不存在 | 展示为可应用，不在应用列表里逐项提示；应用时由系统自动创建目标目录 |
| 项目根路径不存在 | 左侧弱警告，项目应用主区阻断应用，提示到设置修复 |
| 组合为空 | 允许保存，禁用应用，详情提示先编辑成员 |
| 组合成员缺失 | 组合状态为有问题，应用时跳过缺失成员并提示编辑组合 |
| 加载应用状态失败 | 显示可恢复失败态，提供刷新重试和错误详情 |
| 新建组合重名 | 阻断保存，提示换名称或编辑已有组合 |

## 5. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 应用首次打开时能看到非阻塞初始化清单。
- 没有技能、没有项目、没有组合时，页面都给出明确下一步。
- Agent Skill 根目录不存在时，仍允许应用；应用列表不暴露可自愈目录提示。
- 项目根路径不存在时，应用动作被阻断。
- 新建重名技能组合不会覆盖已有组合。
- 加载失败时不是空白屏，能刷新重试。
