# 初始化引导体验优化设计

日期：2026-05-23

## 1. 背景

当前初始化引导把用户带进了一个偏配置表单的流程：小卡片承载长说明，路径、压缩包、插件目录、软链接等内部概念暴露过多。首次打开时，用户还没有建立对「影子」的产品理解，就被要求处理一组技术决策。

本次优化的目标不是做教程，而是让首次体验完成两件事：

- 建立产品身份：影子是本地 Skill 管理专家。
- 让用户轻量完成必要选择：添加 Agent、添加项目、导入已有 Skill、安装应用预置 Skill。

## 2. 核心决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 新增全屏欢迎页 | 当前第一屏直接进入配置，缺少产品信任感和记忆点 | 用户先理解「这是什么」，再进入选择 |
| 欢迎页只保留一个主按钮 | 欢迎页是门面，不应该让用户在第一秒做分支选择 | 操作更果断，第一印象更完整 |
| 后续引导改为全屏工作流 | 卡片套卡片让简单任务显得沉重 | 用户感觉是在轻量整理，而不是填写设置表 |
| 每个环节都可跳过 | 初始化不应阻塞进入工作台 | 用户可以按当前意愿完成，不被流程绑住 |
| 预置 Skill 内化为应用能力 | 压缩包只是工程交付载体，不是用户心智中的来源 | 用户看到的是「应用自带 Skill」，不是「上传文件」 |
| 删除组级全选 / 取消 | 分组只是帮助阅读，不应增加额外操作层级 | 用户只做单个 Skill 的勾选判断 |

## 3. 欢迎页设计

欢迎页是产品门面，但不能做成营销首页。

内容只保留：

- 小尺寸应用 icon。
- 产品名：`影子`。
- 标语：`你的本地 Skill 管理专家`。
- 主按钮：`开始探索`。

不提供次按钮，不放副文案。

### 主视觉

主视觉不使用大 icon、大 `S` 字母或具象 logo 元素。推荐方向是「影子层场」：

- 冷中性浅灰空间。
- 右侧由几层轻薄半透明面、柔和投影和轻微错位组成。
- 表达“一个能力在多个位置留下影子”的隐喻。
- 左侧保留足够留白给标题和按钮。
- 气质关键词：克制、空间感、安静、可信、有一点记忆点。

避免：

- 巨大图标。
- 机器人、网络节点图、架构图。
- 紫色渐变、发光边框、玻璃拟态、赛博风。
- 装饰性光球、泡泡或 bokeh。

## 4. 引导流程

流程顺序调整为：

```text
欢迎页
→ 添加 Agent
→ 添加项目
→ 已有 Skill 导入
→ 预置 Skill 安装
→ 进入工作台
```

这样先确定管理范围，再整理范围内已有内容，最后安装应用自带能力。

| 步骤 | 标题 | 用户任务 | 主按钮 | 次操作 |
|---|---|---|---|---|
| 0 | 欢迎页 | 建立产品理解 | 开始探索 | 无 |
| 1 | 添加 Agent | 选择要管理的工具 | 继续 | 跳过 |
| 2 | 添加项目 | 可选添加常用项目 | 继续 | 跳过 |
| 3 | 已有 Skill 导入 | 选择散落在 Agent 里的 Skill 是否入库 | 导入已选 | 跳过 |
| 4 | 预置 Skill 安装 | 选择应用自带 Skill 是否安装到主库 | 安装已选 | 跳过 |

每个环节都允许用户跳过；需要选择的内容都支持逐项勾选和取消。

## 5. 文案设计

| 页面 | 标题 | 描述 |
|---|---|---|
| 添加 Agent | 添加 Agent | `选择要管理的工具。已检测到的工具会默认选中，也可以先跳过。` |
| 添加项目 | 添加项目 | `把常用项目加入影子，之后可以按项目管理 Skill。也可以稍后再加。` |
| 已有 Skill 导入 | 已有 Skill 导入 | `这些 Skill 已经在你的工具里。导入后会进入主库，方便统一管理。` |
| 预置 Skill 安装 | 预置 Skill 安装 | `影子自带一些常用 Skill。选择你想安装的，之后可以在主库里统一管理。` |

不在主流程出现：

- 压缩包路径。
- `skills` 目录。
- 插件目录。
- 软链接。
- “已忽略插件内 N 个”。

这些信息属于实现细节，只能在必要的错误排查或调试信息中出现。

## 6. 预置 Skill 规则

预置 Skill 是影子应用的一部分，不是用户上传内容。

界面规则：

- 不展示压缩包路径。
- 不提供「选择压缩包」。
- 不使用「压缩包」「预制包」等说法。
- 统一使用「预置 Skill」。
- 顶部只展示用户需要的结果，例如：`找到 14 个预置 Skill，11 个可安装`。
- Skill 按场景分组展示，分组只用于阅读。
- 不提供「全选本组」或「取消本组」。
- 每个 Skill 只保留名称、一句话说明、状态和单项勾选。

状态文案：

| 状态 | 文案 |
|---|---|
| 未安装 | 可安装 |
| 已存在于主库 | 已安装 |
| 读取失败 | 预置 Skill 暂时不可用，可以先跳过。 |

实现上可以继续把当前 zip 作为工程资源来源，但运行时不应暴露给用户。更稳定的方向是把正式 Skill 提取为应用内资源目录，由后端读取应用预置资源。

## 7. 视觉与布局规则

| 模块 | 规则 |
|---|---|
| 整体背景 | 冷中性浅灰，和工作台一致 |
| 欢迎页 | 可以更有品牌感，但仍保持产品工具气质 |
| 后续步骤 | 全屏工作流，不使用居中卡片 |
| 进度 | 左侧或顶部轻量显示，不抢任务区 |
| 列表 | 轻列表行，不做厚卡片和大面积蓝色选中 |
| 路径 | 可作为小号辅助信息展示，但不能成为主信息 |
| 按钮 | 每屏一个主按钮，跳过为次操作 |
| 空状态 | 说明下一步，不只显示“没有数据” |

需要用户操作的内容不要太重：避免大边框、大选中底色、复杂统计和多余动作。

## 8. 验证标准

实现后必须验证：

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- 桌面应用能启动。
- 欢迎页视觉完整，主按钮清晰。
- 四个环节都能跳过。
- Agent、已有 Skill、预置 Skill 都能逐项勾选。
- 预置 Skill 页不再出现压缩包、路径、选择文件等用户不该理解的信息。
- 常见桌面窗口尺寸下不重叠、不溢出。
