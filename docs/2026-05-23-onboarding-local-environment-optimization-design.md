# 初始化引导本地环境优化设计

日期：2026-05-23

## 1. 背景

当前初始化引导已经覆盖欢迎页、Agent、项目、已有 Skill、预置 Skill，但对普通用户的真实本地环境还不够友好：

- 添加项目需要用户自己理解“哪个目录值得纳入管理”。
- 已有 Skill 没有可导入项时，主按钮会失效，流程显得卡住。
- 已有 Skill 扫描只看全局 Agent，容易漏掉刚添加项目里的项目级 Skill。
- 预置 Skill 只提供单个 Skill，没有把外部工具能力按“技能组合”打包。

本次目标是让初始化更像“整理本机现状”，而不是让用户完成一组技术配置。

## 2. 核心决策

| 决策 | 为什么 | 对用户的影响 |
|---|---|---|
| 第二步浅扫描文稿目录并推荐项目 | 用户通常不知道哪些目录符合 Agent 工作目录特征 | 用户可以勾选候选项目，不必从文件选择器里翻找 |
| 只扫描目录，不递归展开所有文件 | 文稿目录可能很大，深扫会慢、吓人，也容易暴露无关内容 | 引导页启动更稳，不会出现一长串无意义文件 |
| 用“项目根 + Agent 痕迹”评分 | 只看 Git 项目会推荐太多开发目录；只看 Agent 目录会漏掉新项目 | 推荐更接近“值得影子管理的工作目录” |
| 第三步空结果时主按钮变为「下一步」 | 空结果是正常状态，不是用户错误 | 用户能顺着流程继续，不会被禁用按钮误导 |
| 第三步扫描全局和项目级存量 Skill | 添加项目后，项目内已有 `.codex/skills`、`.claude/skills` 等也应被发现 | 显著的项目级 Skill 不会被漏掉 |
| 飞书 CLI 作为一个预置技能组合出现 | 官方飞书 CLI 包含多个场景 Skill，逐个展示会增加理解成本 | 用户只看到「飞书技能组」一个选择，安装后在技能组合里统一管理 |
| 失败可跳过，工作台不报错 | 普通用户本机可能没有权限、没有文稿目录、Agent 目录不可读 | 初始化不会因为本地差异失去意义 |

## 3. 第二步：项目候选

新增后端命令 `suggest_onboarding_projects()`，返回少量项目候选，不直接写入数据库。

### 扫描范围

按顺序尝试以下根目录，存在且可读才进入：

- 系统文稿目录：`dirs::document_dir()`。
- 兜底路径：`~/Documents`、`~/文稿`。

扫描规则：

- 只看根目录下的一级子目录。
- 对常见聚合目录可再看一层，例如 `Projects`、`Code`、`Workspace`、`workspace`、`开发`、`项目`。
- 不读取普通文件内容。
- 单个根目录最多检查 200 个目录项，最多返回 8 个候选。
- 跳过隐藏目录、`node_modules`、`target`、`dist`、`build`、`.git`、`Library`、`Applications` 等噪音目录。

### 推荐标准

候选目录按分数排序。推荐不是“确定事实”，而是降低用户找目录的成本。

| 信号 | 加分原因 |
|---|---|
| 存在 `.codex/skills`、`.claude/skills`、`.trae-cn/skills`、`.aiden/skills` | 已经出现 Agent 项目级 Skill 目录，高置信 |
| 存在 `.codex`、`.claude`、`.trae-cn`、`.aiden` | 有 Agent 项目级工作痕迹 |
| 存在 `AGENTS.md`、`CLAUDE.md` | 有 Agent 协作规则 |
| 存在 `.git`、`package.json`、`Cargo.toml`、`pyproject.toml`、`go.mod`、`pnpm-workspace.yaml` | 是项目根目录 |
| 目录名包含 `project`、`workspace`、`repo`、`产品`、`项目` 等 | 弱信号，只用于同分排序 |

进入引导页时，高置信候选默认勾选；低置信候选只展示，不默认勾选。用户点击「继续」时，才把勾选项批量加入项目列表。

## 4. 第三步：已有 Skill 导入

### 空结果主按钮

按钮逻辑调整为：

| 状态 | 主按钮 |
|---|---|
| 正在扫描 | 禁用，显示扫描中 |
| 找到可导入 Skill 且已勾选 | `导入已选（N）` |
| 找到可导入 Skill 但用户取消全部勾选 | `下一步` |
| 没有找到可导入 Skill | `下一步` |
| 扫描失败 | `下一步` |

这一步仍保留「跳过」作为次操作，但主按钮不再因为空结果失效。

### 扫描健全逻辑

现有 `list_unmanaged_for_onboarding()` 扩展为扫描：

- 所有已启用 Agent 的全局 Skill 目录。
- 所有已添加项目下，按已启用 Agent 的 `projectRelativePath` 推导出的项目级 Skill 目录。

扫描仍只识别明显 Skill：

- 目录内存在 `SKILL.md`。
- 或软链接目标存在 `SKILL.md`。

不把只有 `README.md` 的目录当作初始化导入候选，避免把普通文档目录误判为 Skill。

## 5. 第四步：飞书技能组

### 来源

事实：官方 `larksuite/cli` 仓库说明其提供 Lark/Feishu CLI，覆盖 Docs、Sheets、Calendar、Mail、Tasks、Meetings 等业务域，并提供 Agent Skills；截至 2026-05-23，GitHub 页面显示最新 release 为 `v1.0.39`，发布时间为 2026-05-22。

实现时从官方仓库的最新 release 下载 `skills/` 目录，放入应用资源 `src-tauri/resources/preset-skills`。

### 展示

在第 4 步「其他工具」分类里增加一个虚拟项：

- 名称：`飞书技能组`
- 说明：`安装官方飞书 CLI 的 Agent Skills，并创建一个可批量应用的技能组合。`
- 计数：展示成员数量，例如 `26 个成员`

不逐个展示 `lark-doc`、`lark-im`、`lark-sheets` 等成员，避免列表膨胀。

### 安装规则

用户勾选「飞书技能组」后：

1. 把官方 `lark-*` Skill 复制进主库；已存在的同名 Skill 跳过，不覆盖。
2. 创建或更新技能组合：
   - `id`: `feishu-skill-group`
   - `name`: `飞书技能组`
   - `description`: `官方飞书 CLI 技能组合，用于通过 lark-cli 操作文档、消息、表格、日历等飞书能力。`
   - `skillIds`: 所有成功安装或已存在的 `lark-*` Skill。
3. 组合本身不软链接到 Agent 或项目，仍遵守现有技能组合规则。

## 6. 兼容与失败处理

| 场景 | 处理 |
|---|---|
| 文稿目录不存在 | 第二步展示空状态，仍可手动添加项目或继续 |
| 文稿目录不可读 | 跳过该根目录，展示简短提示，不阻断流程 |
| 候选目录已添加过 | 不重复写入，只在列表中显示已添加状态 |
| 项目路径之后失效 | 进入工作台后沿用现有路径不存在提示，不在初始化中阻断 |
| Agent 未安装或目录不存在 | 只扫描已启用且存在的目录；不存在的目标应用时再创建 |
| 飞书资源缺失 | 第四步仍可跳过，不影响进入工作台 |
| 部分飞书 Skill 已存在 | 跳过已有 Skill，仍创建或更新「飞书技能组」 |
| 安装部分失败 | 返回错误摘要，保留进入工作台入口 |

## 7. 实现范围

后端：

- 新增 `ProjectSuggestion` 类型。
- 新增 `suggest_onboarding_projects()` 命令。
- 扩展 `list_unmanaged_for_onboarding()` 扫描项目级 Skill。
- 扩展内置预置扫描，增加「飞书技能组」虚拟项。
- 扩展内置预置安装，支持安装虚拟技能组并写入技能组合。

前端：

- 第二步展示项目候选、勾选状态、推荐理由和手动添加入口。
- 第三步主按钮根据空结果切换为「下一步」。
- 第四步支持预置项的成员数量和技能组文案。

资源：

- 从官方 `larksuite/cli` 最新 release 下载 `skills/lark-*` 目录，作为应用内预置资源。

## 8. 验证标准

- `npm run build`
- `cargo check --manifest-path src-tauri/Cargo.toml`
- `npm run tauri:build`
- 初始化第二步在无候选、无权限、有候选三种情况下都能继续。
- 勾选项目候选后进入第三步，项目级存量 Skill 可以被发现。
- 第三步没有可导入 Skill 时，主按钮显示「下一步」并可点击。
- 第四步「其他工具」出现「飞书技能组」，勾选后安装成员 Skill 并创建技能组合。
- 常见桌面窗口尺寸下，候选列表和预置列表不重叠、不溢出。
