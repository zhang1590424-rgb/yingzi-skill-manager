# Skill 与技能组合详情信息架构设计

日期：2026-05-22

状态：已确认，进入实现

## 1. 背景

当前「技能列表」右侧详情区同时承载了 Skill 描述、主库路径、启用数量、操作按钮、启用位置和 `SKILL.md` 预览。字段本身都有价值，但组织方式偏平铺，且只读路径被做成类似输入框的大块控件，导致用户进入详情后需要自己判断：

- 这个 Skill 当前是否可用？
- 有没有被启用？
- 下一步应该分发、打开目录、删除，还是阅读正文？

「技能组合」详情也有类似问题。当前只展示描述、成员数量和成员名称标签，无法清楚表达组合是否健康、成员是否缺失，以及用户应该编辑还是应用。

本次设计目标不是增加新能力，而是重新组织详情区信息结构，让单个 Skill 和技能组合都围绕用户任务展开。

## 2. 用户场景

### 2.1 单个 Skill 详情

用户进入单个 Skill 详情时，核心问题按优先级排列：

| 优先级 | 用户想看到 | 用户想做 |
|---|---|---|
| 1 | 这个 Skill 是什么，是否正常 | 判断能不能直接分发 |
| 2 | 已经启用了多少位置，有没有问题 | 分发、打开目录、删除 |
| 3 | 启用在哪些对象上 | 确认影响范围 |
| 4 | `SKILL.md` 正文内容 | 需要时阅读能力说明 |
| 5 | 完整文件路径 | 需要定位文件或排查问题时查看 |

### 2.2 技能组合详情

用户进入技能组合详情时，核心问题按优先级排列：

| 优先级 | 用户想看到 | 用户想做 |
|---|---|---|
| 1 | 这个组合解决什么场景，是否可用 | 判断是否能直接应用 |
| 2 | 组合里有哪些成员，成员是否完整 | 编辑成员或处理缺失 |
| 3 | 应用组合会处理什么对象 | 打开应用抽屉 |
| 4 | 组合规则 | 理解组合本身不直接软链接 |

## 3. 核心判断

| 方案 | 判断 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 轻量重排 | 不采用为主方案 | 只调整字段顺序，无法解决组合成员状态不可见的问题 | 界面稍微清楚，但用户仍要自己推断风险 |
| 统一详情框架 | 采用 | Skill 和组合都遵循「摘要 / 状态 / 操作 / 详情」结构 | 进入任何详情页都能先判断状态，再执行动作 |
| 双工作台化 | 暂不采用 | 把 Skill 做成阅读工作台、组合做成编辑工作台会放大第一版复杂度 | 右侧详情会过重，影响三栏工作台效率 |

本次采用「统一详情框架」：

```text
对象标题 + 状态摘要
对象描述
主操作 / 次操作 / 风险操作
概览信息
对象核心明细
二级说明或正文预览
```

## 4. 单个 Skill 详情设计

### 4.1 信息结构

```text
lark-okr                              [未启用] [无问题]
飞书 OKR：管理目标与关键结果...

[分发] [打开目录] [删除]

概览
  格式状态      SKILL.md 正常
  标签          #lark #okr

启用位置
  未启用时：还没有启用位置
  已启用时：Codex / 全局、项目名 / Agent 名

SKILL.md 预览
  默认摘要展示，支持展开 / 收起

更多技术信息
  默认折叠；需要时查看主库路径和目标路径
```

### 4.2 顶部摘要

顶部需要回答「这个 Skill 当前能不能用」。

| 元素 | 规则 |
|---|---|
| 标题 | 使用 `displayName` |
| 描述 | 使用解析后的 `description`，最多保留 2-3 行自然换行 |
| 状态徽标 | 展示启用状态和问题状态，如 `未启用`、`已启用 2 处`、`1 个问题`、`缺少 SKILL.md` |
| 标签 | 不放在标题旁抢占第一注意力，放入概览区 |

### 4.3 操作区

操作区紧跟摘要，避免用户读完正文或技术信息后才能操作。

| 操作 | 类型 | 规则 |
|---|---|---|
| 分发 | 主操作 | 打开分发抽屉 |
| 打开目录 | 次操作 | 打开主库 Skill 目录 |
| 删除 | 风险操作 | 已启用时阻断并提示先收回；未启用时进入确认弹窗 |

每屏仍只保留一个主按钮，符合现有工作台规则。

### 4.4 概览区

概览区放用户日常判断需要的信息，不展示大块只读路径。

| 字段 | 展示方式 | 为什么 |
|---|---|---|
| 格式状态 | 文本 + 状态徽标 | 让 `SKILL.md` 缺失直接可见 |
| 标签 | 小标签行 | 标签是辅助检索信息，不是主操作信息 |
| 主库位置 | 轻量文本，如 `主库内`，旁边提供 `查看路径` | 用户通常只需要知道它已入库，不需要阅读完整路径 |

「主库路径」「启用位置 0 个」都不再作为大输入框样式展示，因为它们不是可编辑字段。

### 4.5 启用位置

启用位置是 Skill 详情的核心状态信息，应该独立成组。

| 状态 | 展示 |
|---|---|
| 0 个启用位置 | 空状态：`还没有启用位置` |
| 有启用位置 | 默认只展示 `Agent / 全局`、`项目 / Agent` 等人能理解的位置名称 |
| 查看完整路径 | 在每个位置行提供轻量 `查看路径` 或折叠区，不默认展开 |
| 后续扩展 | 如果出现冲突、失效、路径不存在，可在该组内显示状态徽标 |

完整路径是定位和排错信息，不是默认管理信息。默认展示路径会让详情区像文件系统审计工具，而不是 Skill 管理工具。

### 4.6 `SKILL.md` 预览

`SKILL.md` 预览保留，但降级为详情底部内容。

| 规则 | 为什么 | 对用户的影响 |
|---|---|---|
| 默认展示摘要高度 | 正文不是每次管理都要阅读 | 分发和状态判断更快 |
| 支持展开 / 收起 | 需要时仍可完整查看 | 不需要跳出应用读文件 |
| 继续使用去 frontmatter 后的正文 | 避免元数据污染阅读 | 内容更接近真实说明 |

## 5. 技能组合详情设计

### 5.1 信息结构

```text
产品经理常用                          [12 个成员] [1 个问题]
用于产品分析、需求拆解、文档整理...

[应用组合] [编辑] [删除]

组合概览
  成员总数      12 个
  可用成员      11 个
  缺失成员      1 个

成员清单
  ✓ lark-okr          飞书 OKR...
  ✓ brainstorming     头脑风暴...
  ! missing-skill     已不在技能列表中

组合规则
  组合本身不会被软链接，应用时只处理成员 Skill。
```

### 5.2 顶部摘要

顶部需要回答「这个组合是否值得应用」。

| 元素 | 规则 |
|---|---|
| 标题 | 使用组合名称 |
| 描述 | 使用组合描述；无描述时显示成员名称摘要 |
| 状态徽标 | 展示成员数量和健康状态，如 `12 个成员`、`1 个缺失`、`空组合` |
| 空组合 | 明确提示不能应用，需要先编辑成员 |

### 5.3 操作区

| 操作 | 类型 | 规则 |
|---|---|---|
| 应用组合 | 主操作 | 有可用成员时打开分发抽屉 |
| 编辑 | 次操作 | 打开组合编辑弹窗 |
| 删除 | 风险操作 | 删除组合配置，不删除主库 Skill |

如果组合没有任何可用成员，`应用组合` 禁用，并提示下一步是编辑成员。

### 5.4 组合概览

组合概览用于让用户快速判断健康度。

| 字段 | 定义 |
|---|---|
| 成员总数 | 组合保存的成员 id 数量 |
| 可用成员 | 当前仍存在于技能列表中的成员数量 |
| 缺失成员 | 成员 id 已不在技能列表中的数量 |
| 异常成员 | 后续可纳入缺少 `SKILL.md`、有 issue 的成员 |

第一版至少表达可用和缺失；如果已有 `hasSkillMd` 与 `issueCount`，可以在成员清单中显示异常。

### 5.5 成员清单

成员清单从标签墙改为列表。

| 成员状态 | 展示 |
|---|---|
| 正常成员 | 名称、描述、可选标签或格式状态 |
| 缺少 `SKILL.md` | 显示警示状态 |
| 有问题 | 显示问题数量 |
| 缺失成员 | 使用保存的 id 展示，并标记 `已不在技能列表中` |

为什么不用标签墙：

- 标签适合少量关键词，不适合承载成员管理。
- 成员需要展示描述和状态。
- 缺失成员如果只是一个标签，风险不明显。

### 5.6 组合规则说明

底部保留一句轻量规则说明：

```text
组合本身不会被软链接；应用或收回组合时，只批量处理成员 Skill。
```

这句话用于降低误解：用户不会以为组合会变成一个独立 Skill 文件夹。

## 6. 共用视觉与交互规则

| 区域 | Skill | 技能组合 |
|---|---|---|
| 顶部 | 名称、描述、启用 / 问题状态 | 名称、描述、成员健康状态 |
| 主操作 | 分发 | 应用组合 |
| 次操作 | 打开目录 | 编辑 |
| 风险操作 | 删除 | 删除 |
| 主体详情 | 概览、启用位置 | 概览、成员清单 |
| 二级内容 | `SKILL.md` 预览 | 组合规则说明 |

视觉规则：

- 保持右侧详情为分组结构，不做卡片堆叠。
- 分组标题使用小标题，不使用大号营销式标题。
- 只读路径不得使用输入框样式。
- 完整路径默认不展开；需要出现时使用轻量折叠区、小号等宽文本或复制/打开动作。
- 状态必须文字 + 颜色共同表达，不只依赖颜色。
- 空状态需要告诉用户下一步，而不是只显示 `0 个`。

## 7. 数据与实现边界

本次只重组前端详情区，不改变底层数据模型。

| 对象 | 使用现有字段 |
|---|---|
| Skill | `displayName`、`description`、`path`、`hasSkillMd`、`tags`、`enabledCount`、`issueCount`、`contentPreview` |
| TargetStatus | `agentName`、`projectName`、`status`、`targetPath`、`issue` |
| Preset | `name`、`description`、`skillIds` |

不新增 SQLite 表，不改变分发、收回、删除的后端规则。

## 8. 错误与边界状态

| 场景 | 处理 |
|---|---|
| Skill 缺少 `SKILL.md` | 顶部显示问题徽标，概览区显示格式异常 |
| Skill 未启用 | 启用位置组展示空状态 |
| Skill 已启用 | 删除按钮仍可见，但点击后阻断并提示先收回 |
| 组合为空 | 顶部显示 `空组合`，应用按钮禁用 |
| 组合有缺失成员 | 顶部显示问题徽标，成员清单中标出缺失项 |
| 组合全部成员缺失 | 应用按钮禁用，只保留编辑和删除 |
| 组合成员有格式异常 | 成员清单显示异常状态，应用前仍走分发预检 |

## 9. 验收标准

设计确认并进入实现后，需要验证：

- 技能列表右侧详情第一屏能看到名称、描述、启用状态、问题状态和主操作。
- `主库路径` 和目标路径不再使用输入框样式。
- 第一屏默认不出现大块完整文件路径；路径进入 `查看路径` 或更多技术信息。
- `启用位置 0 个` 改为空状态表达。
- `SKILL.md` 预览位于状态和操作之后，支持展开 / 收起。
- 技能组合详情不再用标签墙承载成员管理。
- 技能组合详情能显示成员总数、可用成员、缺失成员。
- 缺失成员在成员清单中明确可见。
- 空组合或全部成员缺失时，应用组合不可用，并提示先编辑成员。
- 删除 Skill 和删除组合继续使用确认 / 阻断规则。
- 常见桌面窗口尺寸下，右侧详情不重叠、不溢出。
- `npm run build` 通过。
- `cargo check --manifest-path src-tauri/Cargo.toml` 通过。

## 10. 不做范围

- 不改变主库优先和软链接分发策略。
- 不新增组合嵌套组合。
- 不做组合市场、推荐或模板库。
- 不做覆盖前备份。
- 不做操作日志。
- 不在本轮改造 Agent / 项目应用双栏。
- 不重命名后端 `preset` 数据结构和 SQLite 表。
