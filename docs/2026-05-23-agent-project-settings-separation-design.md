# 设置页 Agent 与项目拆分设计

日期：2026-05-23

## 1. 背景

设置页把「我的 Agent」和「项目」放在同一个「应用位置」区块里。这个抽象在系统实现上成立，因为两者都是 Skill 的应用目标；但在用户心智上会混淆：

- Agent 是全局运行环境，核心信息是全局 Skill 目录和项目目录规则。
- 项目是手动纳入管理的工作空间，核心信息是项目路径是否可访问。

用户在设置页的任务不是理解统一抽象，而是快速判断“我在管理哪些 Agent”“我手动纳入了哪些项目”“技能根目录在哪里”。因此设置页需要拆成三个视觉一致的同级管理区块。

## 2. 核心决策

| 维度 | 决策 | 为什么 | 对用户的影响 |
|---|---|---|---|
| 设置页结构 | 去掉单一「应用位置」大框，拆成「我的 Agent」「我的项目」「技能根目录」三个同级区块 | 统一抽象降低了实现复杂度，但提高了识别成本 | 用户一眼能区分全局 Agent、项目路径和技能主库 |
| Agent 区块 | 保留全局目录、项目目录规则、启用、重新选择、移除 | Agent 的关键风险来自目录识别和启停状态 | 不需要理解项目路径，也能判断 Agent 是否可用 |
| 项目区块 | 只展示手动添加项目、路径状态和移除 | 项目是用户主动纳入管理的范围，不是 Agent 配置的一部分 | 项目列表更像“我的工作空间清单” |
| 技能根目录区块 | 保留主库、数据库和配置文件路径，视觉样式与前两组一致 | 技能主库是事实源，数据库和配置文件是关联定位信息，不应像附注一样贴在页面底部 | 用户能把技能主库识别为第三类设置对象 |

## 3. 界面结构

```text
设置
  健康总览（仅有问题时出现）

  我的 Agent
    添加 Agent
    Trae CN / Codex / Claude Code / Aiden / 手动添加 Agent

  我的项目
    添加项目
    手动添加的项目路径

  技能根目录
    主库路径
    数据库路径
    配置文件路径
```

## 4. 视觉规则

- 「我的 Agent」「我的项目」「技能根目录」都是设置页的一层区块，使用相同面板语言，但中间保留明确间距。
- 三个区块的标题前使用对应图标：Agent 用全局/网络含义图标，项目用文件夹/项目图标，技能根目录用归档/存储含义图标。
- 不使用区块之间的嵌套卡片，不把项目或技能根目录作为 Agent 区块里的子段落。
- 项目为空时，空态只在「我的项目」区块内部出现。

## 5. 实现范围

- 更新设置页 JSX，把 Agent、项目和技能根目录拆成三个同级 section。
- 调整设置页样式，新增同级目标区块样式。
- 不改变 Agent、项目、扫描、软链接分发、收回的数据模型和业务逻辑。

## 6. 验证点

- 设置页不再出现「应用位置」大框。
- 「我的 Agent」「我的项目」「技能根目录」是三个视觉一致的同级面板。
- Agent 的添加入口只在 Agent 面板标题区。
- 项目的添加入口和空态只在项目面板内部。
- 常见桌面窗口尺寸下路径、按钮和空态不重叠、不溢出。
